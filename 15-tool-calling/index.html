Why Tool Calling Exists

A balanced, connected lesson covering why language models need external tools, what standalone LLMs cannot do, and why tools are necessary for calculations, retrieval, and real-world actions.

Part 1Why Tool Calling Exists
Part 2Why an LLM cannot independently access live systems
Part 3Why external tools are required

1. The central idea

⌄

A Large Language Model is mainly designed to understand and generate language. It can explain, summarise, classify, extract information, generate code, and create structured responses.

But many user requests require more than language generation. A user may ask for current inventory, live weather, a private order status, a calculation, or an action such as sending an email.

Tool calling connects an LLM’s language intelligence with external data, calculations, and actions.

2. What a standalone LLM can and cannot do

⌄

A standalone LLM can work with its trained knowledge, conversation context, and information supplied to it.

Without a connected tool, it normally cannot directly read private systems or perform confirmed external operations.

CapabilityStandalone LLM
Explain inventory conceptsYes
Generate SQL for an inventory lookupYes
Know current stock in a private databaseNo
Update stock and confirm successNo
Knowing how an action works is not the same as performing that action.

3. Knowledge is not live information

⌄

If a user asks for the current status of order ORD-2048, the model can recognise the intent and extract the order ID. But the current status must come from the order system. 

User asks for current order status
        ↓
LLM understands the request
        ↓
LLM requests the order-status tool
        ↓
Tool queries the order system
        ↓
Current result is returned
        ↓
LLM explains the result

The tool supplies the live fact. The model turns that fact into a useful human response.

4. Understanding is not execution

⌄

Consider: “Schedule a meeting with Ravi tomorrow at 4 PM.”

The model can extract the action, person, date, and time. But the calendar system must actually:

  1. Find Ravi’s contact details.
  2. Check availability.
  3. Create the event.
  4. Add the attendee.
  5. Save and send the invitation.
  6. Confirm success.
Understanding the action ≠ Performing the action.

5. The basic tool-calling flow

⌄
  1. The user makes a request.
  2. The model recognises that external information or action is required.
  3. The model prepares a structured tool request.
  4. The application validates and executes the tool.
  5. The tool result returns to the model.
  6. The model produces the final natural-language response.
{
  "tool": "get_weather",
  "arguments": {
    "location": "Bengaluru"
  }
}

The model usually requests execution; the surrounding application performs the actual call.

6. Three responsibilities

⌄

LLM

Understands intent, selects a tool, prepares arguments, and explains the result.

Application

Exposes approved tools, validates inputs, checks permissions, executes, and handles errors.

Tool

Calculates, retrieves, reads, searches, updates, or performs the approved external action.

LLM decides. Application controls. Tool executes.

7. Why external tools are required

⌄

A. Calculations

Tools provide deterministic results for tax, finance, statistics, conversions, and large numerical operations.

B. Retrieval

Tools retrieve inventory, orders, files, documents, emails, search results, and database records.

C. Real-world actions

Tools send emails, create events, update orders, raise tickets, cancel bookings, and save documents.

8. Practical retail example

⌄

User: “Do we have 20 units of SKU-501 at the Whitefield store?”

{
  "sku": "SKU-501",
  "store": "Whitefield",
  "required_quantity": 20
}

The inventory tool returns an available quantity of 16.

Final answer: “The Whitefield store has 16 units of SKU-501. It is short by 4 units.”

The model understood the request; the inventory system supplied the fact.

9. Why controlled access matters

⌄

A model should not have unrestricted access to every system or destructive operation.

The application decides:

  • which tools are available,
  • which arguments are valid,
  • which users are authorised,
  • and which actions need confirmation.
Allowed tools:
- get_order_status
- check_inventory
- create_support_ticket
- draft_email

10. Limits, risks, and the agent distinction

⌄

Tool calling improves grounding, but it does not guarantee correctness.

  • The model may choose the wrong tool.
  • Arguments may be extracted incorrectly.
  • The tool may return stale or incomplete information.
  • The final answer may misinterpret the result.

A single tool call also does not automatically make a system an agent.

Tool calling = access to capabilities.
Agent = a decision-making loop that may use capabilities repeatedly.

11. Step 1 recap

⌄

A standalone LLM is powerful at language but does not automatically have live access to external systems.

User expresses the request
        ↓
LLM understands and prepares
        ↓
Application validates and controls
        ↓
Tool performs the operation
        ↓
LLM explains the confirmed result
The LLM understands and proposes. The application controls. The tool performs.

12. Mini quiz

⌄
1. Can a standalone LLM know the latest stock quantity in a private database?

No. It requires an approved database or inventory tool.

2. What is the difference between understanding and execution?

The LLM may understand the request, but external code and tools must perform the real operation.

3. What are the three main reasons tools are needed?

Calculation, retrieval, and real-world action.

4. Does one tool call automatically create an agent?

No. An agent normally includes repeated planning, observation, and decision-making.

5. Who controls whether a tool is allowed to run?

The surrounding application or orchestration layer.

Tools, Functions, APIs, and External Capabilities

Understand how ordinary application code becomes a controlled model-facing capability, and how functions, APIs, services, and workflows fit together in a tool-calling system.

1. From language model to useful application

⌄

An LLM can understand a request, but the capability needed to fulfil that request may live elsewhere: inside application code, a database, an API, a search service, or a business workflow.

A tool is a capability made available to the model under application control.

2. What is a tool?

⌄

A tool is an approved operation that the model can request.

get_weather
calculate_tax
check_inventory
search_documents
create_calendar_event
send_email
get_order_status
create_support_ticket

A tool normally includes:

  1. A name
  2. A description
  3. Input parameters
  4. An implementation
  5. A result

3. Tool as a model-facing concept

⌄

The model sees a simple capability such as get_order_status. The application may implement it using a local method, REST API, stored procedure, microservice, or a larger workflow. 

Model sees:
get_customer_balance

Application may use:
C# method
    ↓
Banking service API
    ↓
Database
Model sees the tool. Application owns the implementation.

4. What is a function?

⌄

A function is a reusable block of program logic that performs a specific task.

Python

def calculate_total(price, quantity):
    return price * quantity

JavaScript

function calculateTotal(price, quantity) {
  return price * quantity;
}

C#

decimal CalculateTotal(decimal price, int quantity)
{
    return price * quantity;
}

A function does not automatically become available to an LLM merely because it exists in the codebase.

5. How a function becomes a tool

⌄

The application must describe, register, and expose the function.

{
  "name": "get_order_status",
  "description": "Retrieves the current status of a customer order.",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "The unique order identifier."
      }
    },
    "required": ["order_id"]
  }
}
Tool definition
       ↓
Model selects tool
       ↓
Structured arguments
       ↓
Application finds matching function
       ↓
Function executes

7. What is an API?

⌄

An API is a defined interface through which one software system communicates with another. 

GET /weather?city=Bengaluru
{
  "city": "Bengaluru",
  "temperature_c": 27,
  "condition": "Cloudy"
}

An API contract defines the available operation, inputs, authentication, request format, and response.

8. How an API becomes an AI tool

⌄

Application code wraps an API in a controlled tool rather than giving the model unrestricted network access.

def get_weather(city):
    response = weather_api.get(
        "/weather",
        params={"city": city}
    )
    return response.json()
User request
    ↓
LLM calls get_weather
    ↓
Application validates city
    ↓
Application calls weather API
    ↓
Result returns to LLM
    ↓
LLM explains result

API URLs, keys, authentication headers, and retries stay inside the application.

10. External capabilities

⌄

Live information

Weather, news, traffic, prices, shipment tracking, and inventory.

Private information

Orders, company documents, emails, events, records, and tickets.

Deterministic computation

Tax, currency conversion, statistics, optimisation, and calculations.

Real-world actions

Send, create, update, submit, cancel, save, or trigger a workflow.

11. Tools extend the LLM beyond text

⌄

Without tools, the model mainly explains what could be done.

With tools, the system can check, calculate, retrieve, or perform an approved action and return a confirmed result.

The LLM is not the database, calculator, or email system. It is the intelligent interface that knows when and how to use them.

12. One request may involve many implementation layers

⌄

Example: “Check SKU-501 and create a replenishment request if stock is below 20.”

LLM
 ↓
Tool definition
 ↓
Application dispatcher
 ↓
C# service method
 ↓
Inventory REST API
 ↓
Database

A simple tool interface can hide a complex enterprise implementation.

13. Tool abstraction

⌄

Abstraction shows the necessary interface while hiding unnecessary internal complexity.

{
  "sku": "SKU-501",
  "store": "Whitefield"
}

The model does not need database connection strings, SQL joins, service URLs, access tokens, retries, or caching rules.

14. Business-system example

⌄

For “Where is my order ORD-9001?”, the tool may validate ownership, query order management, query a carrier, combine results, and return safe data.

{
  "order_id": "ORD-9001",
  "order_status": "Shipped",
  "carrier": "BlueDart",
  "expected_delivery": "2026-07-12"
}

One simple tool can represent a sophisticated business capability.

15. Intentional exposure and tool granularity

⌄

Not every application method should become a model tool.

Good tools are limited, purposeful, well described, permission controlled, validated, and auditable.

Too small

May require many calls and increase complexity.

Too large

May be difficult to control, test, and reuse.

Aim for one clear business capability with well-defined inputs and predictable output.

16. Function calling and tool calling terminology

⌄

Function calling focuses on structured requests to functions. Tool calling is broader.

Function calling ⊂ Tool calling

A tool may represent a function, API, database operation, search engine, file system, or business workflow.

17. Complete mental model

⌄ 
External capability
        ↓
Implemented through function, API, service, or workflow
        ↓
Wrapped and registered by the application
        ↓
Presented to the model as a tool
        ↓
Requested through structured tool/function calling
The model sees a simple tool interface. The application hides and controls implementation complexity.

18. Mini quiz

⌄
1. Is every function automatically an LLM tool?

No. It must be intentionally described, registered, and exposed.

2. Can one tool internally call several functions or APIs?

Yes. A tool may represent a larger workflow.

3. What is the main difference between a function and a tool?

A function is application code; a tool is a capability exposed to the model.

4. Why should API keys stay out of tool definitions?

They are secrets and must remain protected inside the application.

5. What is an external capability?

Live data, private information, deterministic computation, system interaction, or real-world action.

6. Which is broader: function calling or tool calling?

Tool calling is broader.

3. One complete order-status cycle

⌄

User: “What is the current status of order ORD-2048?”

  1. The application receives the request.
  2. It sends the request and available tool definitions to the LLM.
  3. The LLM recognises that current order data is required.
  4. The LLM requests get_order_status.
  5. The application validates the request.
  6. The tool queries the order-management system.
  7. The result returns to the application.
  8. The application sends the result back to the LLM.
  9. The LLM prepares a user-friendly answer.

4. The user request enters the application

⌄

The request may enter through a website, mobile app, chatbot, support portal, Teams assistant, or enterprise application.

The application may add:

  • User identity
  • Conversation history
  • Available tools
  • System instructions
  • Permissions and organisational rules

5. The LLM understands and requests

⌄ 
Intent: Retrieve order status
Order ID: ORD-2048
External information required: Yes
Selected tool: get_order_status
{
  "name": "get_order_status",
  "arguments": {
    "order_id": "ORD-2048"
  }
}
This is a request for execution, not the execution itself.

6. Application validation before execution

⌄

The application should check:

  • Does the requested tool exist?
  • Is it enabled?
  • Is the user authorised?
  • Are all required arguments present?
  • Are the argument types and values valid?
  • Does the action need confirmation?
  • Has a rate limit been exceeded?
Model output must never be treated as automatically trusted application code.

7. Application orchestration

⌄

The coordinating component may be called an orchestrator, dispatcher, controller, tool runner, agent runtime, or execution loop.

8. What the orchestrator controls

⌄

Registration

Maintains the approved tool list.

Routing

Maps requested tool names to real functions.

Validation

Checks schema, values, and required inputs.

Authorisation

Checks whether the user may perform the action.

Execution

Calls the function, API, database, or workflow.

Loop control

Determines whether more tool calls are needed.

9. The execution boundary

⌄

LLM

Understands, selects, extracts, interprets, and explains.

Application

Defines, validates, authorises, executes, logs, and controls.

Tool

Queries, calculates, reads, writes, or performs the approved operation.

LLM decides what should be requested. Application decides whether and how it may run. Tool performs the operation.

10. Why the boundary is necessary

⌄

A refund request should never be executed merely because the model produced structured JSON.

{
  "name": "refund_payment",
  "arguments": {
    "order_id": "ORD-2048",
    "amount": 50000
  }
}

The application must verify ownership, refund eligibility, amount, approval requirements, and duplicate execution.

18. Simple orchestration pseudocode

⌄ 
response = model.ask(
    user_message=user_message,
    tools=available_tools
)

if response.requests_tool:
    tool_name = response.tool_name
    arguments = response.arguments

    validate(tool_name, arguments)
    tool_result = execute(tool_name, arguments)

    final_response = model.ask(
        user_message=user_message,
        tool_result=tool_result
    )
else:
    final_response = response

return final_response

19. Common misunderstandings

⌄
  • “The LLM directly called the API.” Usually, the application called it.
  • “The model executed the function.” Usually, the model requested it.
  • “The tool result is the final answer.” Often it is returned to the model first.
  • “Orchestration is optional.” Some coordination layer is always needed.
  • “Tool output is always trustworthy.” It may fail, be stale, incomplete, or unsafe.

1. Why a tool needs a definition

⌄

A function in application code is not automatically understandable or safely callable by an LLM. The application must explain what it does, when it should be used, and what inputs it expects.

A tool definition is the contract between the model and the application.

2. Main parts of a function definition

⌄
  1. Tool or function name
  2. Description
  3. Parameter schema
  4. Individual parameter definitions
  5. Required-field list
  6. Optional constraints
{
  "name": "check_inventory",
  "description": "Checks the available quantity of an item at a store.",
  "parameters": {
    "type": "object",
    "properties": {
      "sku": {
        "type": "string"
      },
      "store_id": {
        "type": "string"
      }
    },
    "required": ["sku", "store_id"]
  }
}

3. Choosing a good function name

⌄

Good

get_order_status
check_store_inventory
create_calendar_event
cancel_hotel_booking

Weak

process
execute
handle_data
do_action

Names should be clear, specific, action-oriented, and easy to distinguish from other tools.

4. Name the business action

⌄ 
Implementation-focused:
execute_order_database_query

Business-focused:
get_order_status
The model should understand the user-facing capability, not the hidden implementation detail.

5. Writing a strong description

⌄

A strong description explains what the tool does, when to use it, and sometimes what it does not do.

{
  "name": "get_order_status",
  "description": "Use this tool when the user asks for the current processing, shipping, or delivery status of an existing order. An order ID is required."
}

This is stronger than simply saying: "Gets order information."

6. Weak versus strong descriptions

⌄

Weak

"Searches orders."

Strong

"Searches existing customer orders using filters such as customer ID, status, store, or creation date. This tool only retrieves orders and does not modify them."

7. Parameters

⌄

Parameters define the inputs required by a tool.

{
  "sku": "SKU-501",
  "store_id": "WHITEFIELD-01",
  "required_quantity": 20
}

Each parameter should define its name, type, purpose, requirement status, and constraints.

8. Unambiguous parameter names

⌄ 
Weak:
{ "id": "2048" }

Better:
{ "order_id": "ORD-2048" }

Precise names reduce argument-extraction errors.

9. JSON data versus JSON Schema

⌄

Actual JSON data

{
  "sku": "SKU-501",
  "quantity": 20
}

JSON Schema

{
  "type": "object",
  "properties": {
    "sku": { "type": "string" },
    "quantity": { "type": "integer" }
  },
  "required": ["sku", "quantity"]
}
JSON contains values. JSON Schema defines the rules for those values.

    "required_quantity": { "type": "integer" }
  },
  "required": ["sku", "store_id"]
}

Appearing under properties does not automatically make a field required.

}

  "type": "string",
  "minLength": 1
}
A value can be present and still be invalid.

async def run_tool_loop(
    model_client,
    messages,
    tool_definitions,
    dispatcher,
    max_iterations=8
):
    for _ in range(max_iterations):
        response = await model_client.create_response(
            messages=messages,
            tools=tool_definitions
        )

        messages.append(response.assistant_message)

        if not response.tool_calls:
            return response.final_text

        for call in response.tool_calls:
            arguments = parse_arguments(call.arguments)

            result = await dispatcher.dispatch(
                tool_name=call.name,
                arguments=arguments,
                tool_call_id=call.id
            )

            messages.append({
                "role": "tool",
                "tool_call_id": call.id,
                "name": call.name,
                "content": result
            })

    raise RuntimeError("Maximum iterations reached.")

async function runToolLoop({
  modelClient,
  messages,
  toolDefinitions,
  maxIterations = 8
}) {
  for (let i = 0; i < maxIterations; i++) {
    const response = await modelClient.respond({
      messages,
      tools: toolDefinitions
    });

    messages.push(response.assistantMessage);

    if (!response.toolCalls?.length) {
      return response.finalText;
    }

    for (const call of response.toolCalls) {
      const args = parseArguments(call.arguments);

      messages.push({
        role: "tool",
        tool_call_id: call.id,
        name: call.name,
        content: JSON.stringify(result)

}

49. Golden rules

⌄
  1. The model proposes; the application controls.
  2. Validate every argument.
  3. Never treat memory as permission.
  4. Separate read, write, and destructive tools.
  5. Confirm meaningful side effects.
  6. Use approval for high-impact decisions.
  7. Make writes idempotent.
  8. Treat retrieved content and outputs as untrusted data.
  9. Test selection and arguments separately.
  10. Observe important actions.
  11. Use workflows for predictable high-risk processes.
  12. Give agents only the tools they need.
  13. Never claim completion before verified success.

50. Final mastery quiz

⌄
1. What is the model's role?

Interpret language and propose structured requests for approved capabilities.

2. Who executes the real function?

The application or connected tool server.

3. Why is JSON parsing insufficient?

Parsed values can still violate types, rules, permission, or confirmation requirements.

4. What is idempotency?

Protection against duplicate real-world effects for repeated intended operations.

5. What is MCP?

A standard protocol connecting AI hosts to capability servers.

6. Workflow versus agent?

Use workflows for known governed processes and agents for open-ended investigation.

7. Why test no-tool decisions?

To prevent unnecessary external calls, cost, latency, and exposure.

8. What should happen when information is missing?

Ask the user rather than inventing it.

9. What should the assistant say for a pending operation?

That it is pending, not completed.
