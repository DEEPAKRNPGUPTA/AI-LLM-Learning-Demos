Transformer

A deep single-page study guide explaining how Transformers turn tokens into vectors, update meaning through attention and neural networks, and predict the next token.

Core idea

A Transformer is a neural network architecture that uses self-attention, feed-forward networks, residual connections, and normalization to repeatedly improve token representations.

Memory line: Attention is the mechanism. Transformer is the architecture. GPT is a decoder-only Transformer trained for next-token prediction.

Contents

Before Transformers, sequence learning mainly used RNNs, LSTMs, and GRUs. These models process tokens one after another.

word 1 → word 2 → word 3 → word 4

This sequence-by-sequence flow creates a memory bottleneck. Information from early tokens must travel through many hidden states before it reaches later tokens.

RNN-style thinking

I remember what came before through one moving memory.

Transformer-style thinking

I compare every token with every other token and build direct relationships.

Transformers helped because tokens can directly attend to other tokens. This makes long-distance relationships easier and training more parallelizable on GPUs.

Remember: RNN carries memory step by step. Transformer lets tokens directly talk to each other.

Why this became a scaling problem

RNN-style models were meaningful historically, but large-scale language learning needs massive parallel training. A Transformer can compare many token relationships using matrix operations, which works much better with GPUs.

The important shift is not only speed. The shift is representation. A token can directly collect information from another token without waiting for hidden state memory to carry it forward.

A Transformer is built by stacking repeated blocks. Each block improves the representation of every token.

The Transformer does not read raw text directly. It works with numeric vectors. The architecture repeatedly updates those vectors until they become context-rich.

PartPurpose
EmbeddingStarting numeric meaning of token
PositionOrder information
AttentionTokens collect information from other tokens
FFNEach token processes its internal representation
Output layerConverts hidden vectors into token scores

Repeated block idea

One Transformer block gives one round of improvement. Many blocks create many rounds of token communication and transformation. Early layers may learn local grammar-like patterns, while deeper layers may represent richer meaning and task-relevant structure.

Text first becomes tokens. Tokens become token IDs. Token IDs become embeddings.

Text → Tokens → Token IDs → Embeddings

Example:

"I love machine learning"
→ ["I", "love", "machine", "learning"]
→ [40, 1842, 6221, 3150]
→ vectors from embedding table

A token ID is only an index. It tells the model which row to pick from the embedding table. The embedding vector is the starting meaning.

Important: Initial embedding is not final meaning. Transformer layers turn it into contextual meaning.

Example: the token bank may start with the same embedding in both sentences, but after attention it becomes different:

I deposited money in the bank.     → financial meaning
We sat near the river bank.        → river-side meaning

Static vs contextual meaning

The embedding table gives a starting vector for a token, but the Transformer changes that vector based on surrounding tokens. This is why the same word can behave differently in different sentences.

For learning, keep this difference clear: token ID is just an index, embedding is starting meaning, and Transformer output is contextual meaning.

Transformer processes tokens together, so it needs explicit position information.

Dog bites man.
Man bites dog.

Same words, different order, different meaning. Position tells the model where each token appears.

final input vector = token embedding + position information

There are different position methods. Classic Transformers used sinusoidal positional encoding. Many models use learned position embeddings. Modern LLMs may use RoPE or other methods.

Memory line: Token embedding tells what the token is. Position tells where the token is.

Absolute and relative position

Some methods tell the model the absolute position, like position 1 or position 2. Other methods help the model understand relative distance, such as one token being three places before another. Modern models care a lot about this because longer context windows need strong position handling.

Self-attention means tokens in the same sequence look at each other.

The cat drank milk because it was hungry.

For it, the model may attend to cat. For drank, the model may attend to cat and milk.

Attention calculates scores, turns them into weights, and uses those weights to mix value vectors.

ConceptSimple meaning
Attention scoreHow strongly one token matches another
Attention weightNormalized importance, often sums to 1 per row
Weighted mixtureNew token vector created from useful token information
new drank = 0.05×The + 0.40×cat + 0.15×drank + 0.40×milk

Context mixing

Attention is a context-mixing operation. It does not simply select one word. It creates a weighted mixture of value vectors, so each token representation becomes a blend of useful information from the sequence.

Self-attention creates three learned versions of each token vector: Query, Key, and Value.

Q/K/VIntuition
QueryWhat am I looking for?
KeyWhat do I contain for matching?
ValueWhat information do I provide?
Attention(Q, K, V) = softmax(QKᵀ / √dₖ) V

Q and K decide how much attention. V provides the actual information that gets mixed.

Memory line: Q and K produce weights. Weights mix V.

Why three views?

The same token needs different roles. As a query it asks for information. As a key it allows other tokens to match with it. As a value it supplies information. Separate learned projections make this flexible.

For 4 tokens, the attention matrix is 4 × 4. Each row says how one token attends to all tokens.

           The   cat   drank   milk
The        .40   .30    .20    .10
cat        .10   .30    .40    .20
drank      .05   .40    .15    .40
milk       .05   .30    .45    .20

In decoder-only models, attention is causal/masked. A token can look only at itself and previous tokens, not future tokens.

              I   love   machine   learning
I             ✅    ❌       ❌         ❌
love          ✅    ✅       ❌         ❌
machine       ✅    ✅       ✅         ❌
learning      ✅    ✅       ✅         ✅
Causal mask: prevents future-token cheating during next-token prediction.

Two mask families

A padding mask ignores fake padding tokens used for batching. A causal mask blocks future tokens in decoder generation. In real systems, both kinds of masks may be combined.

One attention head gives one learned view. Multi-head attention gives several views in parallel.

Input token vectors
   ↓
Head 1 attention
Head 2 attention
Head 3 attention
Head 4 attention
   ↓
Concatenate + linear projection
   ↓
Multi-head output

Different heads may focus on different relationships: pronoun reference, subject-object relation, phrase structure, or cause-effect clues.

Memory line: Multi-head attention gives the Transformer several attention lenses at once.

Many lenses, one representation

Each head works in its own learned subspace. After all heads run, their outputs are concatenated and projected back to the model dimension. This is why the next layer can receive the same shape as the previous layer.

After attention, every token has collected context. The Feed Forward Network processes each token internally.

Attention

Tokens talk to other tokens.

FFN / MLP

Each token thinks internally.

Input vector
  ↓
Linear layer 1
  ↓
Activation
  ↓
Linear layer 2
  ↓
Output vector

Often the FFN expands then compresses:

768 → 3072 → 768

Modern LLMs may use activations such as GELU or SwiGLU.

Where much capacity lives

In many Transformer models, the feed-forward layers contain a large portion of parameters. Attention routes information between tokens; FFNs transform and store many learned patterns inside token representations.

Residual connections preserve old information and add learned updates.

output = input + transformation(input)

Instead of rebuilding the whole vector from scratch, the model learns corrections.

Memory line: Do not erase. Improve.

This helps information and gradients flow through very deep Transformer stacks.

Gradient flow

Residual connections also help backpropagation. The learning signal can travel through skip paths more easily, which makes deep networks train more reliably.

Layer normalization stabilizes the values inside each token vector. It keeps signals from becoming too large, too small, or uneven.

normalized value = (value - mean) / standard deviation

You often see Add & Norm in diagrams:

Add = residual connection
Norm = layer normalization

Classic diagrams often show post-norm. Many modern LLMs use pre-norm because it is more stable for very deep training.

Memory line: Residual preserves. LayerNorm stabilizes.

Pre-norm note

For beginner diagrams, Add & Norm after each sublayer is easy to understand. In many modern LLMs, LayerNorm often happens before attention and before FFN. The goal is the same: keep values stable while allowing deep stacking.

A Transformer block has two main jobs:

StageJob
AttentionCollect useful information from other tokens
FFNTransform each token’s updated information
Memory line: Transformer block = tokens talk, then tokens think.

What stays same and what changes

A block usually keeps sequence length and vector dimension the same. It does not remove tokens. It improves the meaning inside each token vector. This is why blocks can be stacked repeatedly.

An encoder reads the full input and builds understanding. Encoder self-attention is usually bidirectional: tokens can look left and right.

The bank was flooded after heavy rain.

To understand bank, the encoder can look at future words like flooded and rain.

Encoder-only models are good for classification, search, embeddings, similarity, and document understanding.

Memory line: Encoder = understanding machine.

Encoder use cases

Encoder-only models are especially useful when you need a representation of the input: classification, semantic search, sentence similarity, document embeddings, and extraction-style tasks.

A decoder generates output tokens one at a time. It uses masked self-attention so it cannot look at future tokens.

Prompt: The cat drank
Prediction: milk

For GPT-style decoder-only models, the core block is:

Input vectors
  ↓
Masked multi-head self-attention
  ↓
Add & Norm
  ↓
Feed Forward Network
  ↓
Add & Norm
  ↓
Output vectors
Memory line: Decoder = generation machine with future masking.

Decoder logic

The decoder is built to match writing. When writing the next word, you can only use what has already been written. This is why future masking is not a weakness; it matches the generation task.

The original Transformer used both encoder and decoder, especially for translation and sequence-to-sequence tasks.

Source input → Encoder → Cross-attention → Decoder → Target output

It uses three attention types:

  1. Encoder self-attention
  2. Decoder masked self-attention
  3. Encoder-decoder cross-attention

In cross-attention, decoder queries attend to encoder keys and values.

Memory line: Encoder-decoder Transformer reads one sequence and generates another.

Cross-attention intuition

In translation, the encoder reads the source sentence once. While the decoder writes the target sentence, cross-attention lets it look back at the relevant source tokens for each generated token.

GPT-style models are decoder-only Transformers. They treat everything as text continuation.

Prompt tokens
  ↓
Decoder blocks with causal attention
  ↓
Next-token prediction
  ↓
Append token
  ↓
Repeat

This is flexible because many tasks can be written as prompt completion: translation, summarization, classification, code generation, question answering, and chat.

Memory line: GPT = decoder-only Transformer trained for next-token prediction.

Why GPT can do many tasks

Decoder-only models can treat many tasks as text continuation. A prompt such as “Translate this”, “Summarize this”, or “Write code for this” becomes context, and the answer becomes the continuation.

TopicTrainingInference
GoalLearn weightsUse fixed weights
Uses backprop?YesNo
InputTraining textUser prompt
OutputLoss + weight updatesGenerated tokens

Training changes the model. Inference uses the trained model to produce output.

Prompt context is temporary information; it does not normally change the model’s weights.

Prompt context is temporary

During inference, the model can use information from the current prompt, but that does not automatically update model weights. Training and fine-tuning change weights; prompting changes only the current context.

A GPT-style Transformer predicts the next token from previous tokens.

Context: The cat drank
Possible next tokens:
milk   0.45
water  0.22
slowly 0.08

Long answers are built by repeating:

predict → append → predict → append

The model outputs logits first. Softmax converts logits into probabilities. A decoding strategy chooses the next token.

Truth warning: Next-token prediction optimizes plausible continuation, not guaranteed truth. That is why verification, RAG, tools, and citations matter.

Probabilities, not certainty

The model outputs a probability distribution over tokens. Decoding choices like greedy decoding, sampling, temperature, top-k, and top-p decide how those probabilities become actual generated text.

Padding mask vs causal mask

MaskPurpose
Padding maskIgnore fake [PAD] tokens added for batching
Causal maskPrevent attention to future tokens during generation

Cross-entropy loss

Training checks how much probability the model assigned to the correct next token.

loss = -log(probability of correct token)

Teacher forcing / shifted labels

Input:  The   cat   drank   milk
Target: cat   drank milk    .

During training, the model gets correct previous tokens from data. During inference, it uses its own generated tokens.

KV cache

During inference, previous Keys and Values are cached so the model does not recalculate them for every new token.

previous K,V saved → compute only new token K,V → faster generation

Attention cost and context window

Attention compares tokens with tokens, so the attention matrix is n × n. Cost grows roughly as O(n²).

1,000 tokens  → 1,000,000 attention values
10,000 tokens → 100,000,000 attention values

Transformer vs LLM

TransformerLLM
ArchitectureLarge trained language system
Attention, FFN, blocksArchitecture + data + parameters + tokenizer + tuning + serving

All GPT-style LLMs use Transformers, but a Transformer architecture alone is not automatically a useful LLM.

Modern LLM notes

  • RoPE: rotary positional embedding; injects position into Q/K rotations.
  • ALiBi: adds distance-based attention bias.
  • GELU/SwiGLU: activation/gating choices inside FFN.
  • Pre-norm: LayerNorm before attention/FFN, often used for deep stability.
  • Dropout: training regularization used in classic Transformer training.

Why these details matter

These implementation-level ideas explain why Transformers are powerful but expensive, why generation benefits from KV cache, why long context costs more, and why a Transformer architecture alone is not the same thing as a fully trained LLM system.
