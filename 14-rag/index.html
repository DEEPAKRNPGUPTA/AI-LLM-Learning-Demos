Part 1

Detailed Teaching Content

1. What problem RAG solves

A normal LLM answers from what it already learned during training plus what you type in the prompt.

But in real projects, this is not enough.

Example:

You ask:

What is our company leave policy for 2026?

The LLM may know general HR ideas, but it does not automatically know your company’s latest leave policy document.

So RAG solves this problem:

LLM does not know the needed information ↓ System retrieves the right information from documents ↓ LLM answers using that retrieved information

RAG means:

Retrieval-Augmented Generation

That means:

Retrieve useful knowledge first, then generate the answer.
Teaching note: RAG is not about making the LLM magically know everything. It is about giving the model the right reference material before it answers.

2. Why LLMs cannot know everything

An LLM is powerful, but it has limits.

It may not know:

Your company documents Your latest project design Your internal emails Your client contracts Your updated policies Your private training material Your newest database records

Think of an LLM like a very intelligent person who has read many books, but who has not entered your office file room.

RAG gives the LLM access to that file room.

Simple analogy: The LLM is intelligent, but it still needs access to your files. RAG opens the file room, finds the right page, and gives it to the LLM before answering.

3. Knowledge cutoff problem

LLMs are trained up to a certain point in time.

So if something changed after that, the model may not know it.

Example:

Old policy: Work from home allowed 3 days/week. New policy: Work from home allowed 1 day/week.

A normal LLM may answer based on old general knowledge.

A RAG system can retrieve the latest policy document and answer from that.

So RAG helps with:

latest information changing information company-specific information version-specific information
Important: RAG helps with freshness only if the document index is also fresh. If the vector database still contains old chunks, the answer may still be old.

4. Hallucination problem

Hallucination means the model gives an answer that sounds confident but may be wrong.

Example:

User asks:

What is the refund rule in our product manual?

Without documents, the LLM may say:

Customers can request a refund within 30 days.

But maybe the actual document says:

Refunds are allowed only within 7 days.

RAG reduces hallucination because the model is asked to answer using retrieved source content.

So the goal is:

Do not guess. Look up the source. Answer from the source.
Important: RAG reduces hallucination, but does not automatically remove it. Wrong retrieval, bad chunking, weak prompting, or old documents can still create wrong answers.

5. Private/company data problem

Most useful business knowledge is private.

Examples:

HR policies Legal contracts Support tickets SOP documents Training PDFs Architecture documents Internal FAQs Client onboarding guides Project documents

These are not inside the public training data of the LLM.

RAG allows a company to connect its private data to the LLM without retraining the whole model.

This is why RAG is very important in enterprise AI.

Enterprise point: Private data must be connected securely. Later, when we study Enterprise RAG, we will add access control, permissions, audit logs, PII protection, and versioning.

6. Why fine-tuning is not always the answer

Many people think:

If the LLM does not know something, fine-tune it.

But fine-tuning is not always the right solution.

Fine-tuning is better for changing model behavior, style, format, or domain pattern.

RAG is better for giving the model fresh and specific knowledge.

Example:

NeedBetter Approach
Answer from latest policy documentRAG
Answer from company product manualRAG
Follow a special writing styleFine-tuning
Classify support tickets in a fixed formatFine-tuning or prompt design
Use private documents with citationsRAG
Update answers when document changesRAG

Why?

Because if a policy changes every month, fine-tuning again and again is costly and slow.

In RAG, we update the document/index, not the model.

Decision rule: Fine-tuning changes behavior. RAG supplies reference material at answer time.

7. RAG vs normal prompting

Normal prompting:

User question → LLM → Answer

RAG prompting:

User question ↓ Retrieve relevant documents ↓ Add retrieved context to prompt ↓ LLM answers using that context

Normal prompting depends mainly on the LLM’s memory.

RAG depends on:

LLM intelligence + retrieved knowledge

So RAG is more useful when the answer must come from specific documents.

Remember: Prompting can improve instructions, but it cannot create access to missing private or latest documents.

8. RAG vs fine-tuning

Fine-tuning changes the model.

RAG changes what information is supplied to the model.

Simple difference:

Fine-tuning = teach the model a behavior RAG = give the model reference material

Example:

If you want the model to always answer in your company’s tone, fine-tuning may help.

If you want the model to answer from 10,000 company PDFs, RAG is usually the better choice.

Practical memory: RAG is usually better when the knowledge changes or must be cited. Fine-tuning is better when the response behavior itself must change.

9. RAG vs search engine

A search engine finds documents.

RAG finds documents and then uses the LLM to produce an answer.

Search engine:

Question → list of links/documents

RAG:

Question → relevant chunks → LLM-generated answer with sources

Example:

Search result may say:

Here are 10 documents about leave policy.

RAG answer may say:

According to the 2026 HR policy, employees are allowed 18 annual leave days, and carry-forward is limited to 5 days. Source: HR Policy 2026, section 4.2

So RAG is like:

Search + reading + summarizing + answering
Key point: RAG does not simply return documents. It reads retrieved evidence and produces a direct answer.

Detailed Teaching Content

11. Meaning of Retrieval-Augmented Generation

RAG has three words:

Retrieval + Augmented + Generation

Let us break it.

Retrieval

Retrieval means:

Find the relevant information from documents, database, knowledge base, or search index.

Example:

User asks:

What is the refund policy for premium users?

The system searches the company documents and finds the relevant policy section.

Augmented

Augmented means:

The LLM is given extra supporting information before answering.

The LLM is not answering only from memory.

It is answering with added reference material.

Generation

Generation means:

The LLM writes the final human-friendly answer.

So RAG means:

Find the right information first, add it to the prompt, then ask the LLM to generate the answer.
Teaching note: RAG is not just a better prompt. It is a pattern where retrieval supplies evidence and generation turns that evidence into a useful answer.

12. Simple RAG Flow

A simple RAG flow looks like this:

User Question ↓ Search / Retrieve relevant document chunks ↓ Add those chunks to the prompt ↓ Send prompt to LLM ↓ LLM generates answer ↓ Show answer with sources

Example:

User asks:

How many casual leaves are allowed?

RAG system retrieves this chunk:

HR Policy 2026: Employees are allowed 12 casual leaves per calendar year. Unused casual leaves cannot be carried forward.

Then the LLM answers:

Employees are allowed 12 casual leaves per calendar year. Unused casual leaves cannot be carried forward. Source: HR Policy 2026

This is RAG.

Key idea: In a normal chatbot, the LLM answers directly. In RAG, the system first finds useful content and then asks the LLM to answer using that content.

13. User Question → Retrieve Documents → Send Context to LLM → Answer

This is the heart of RAG.

Question → Retrieval → Context → LLM → Answer

Let us understand each stage.

Stage 1 — User Question

User asks something in normal language.

What is the notice period for resignation?

Stage 2 — Retrieve Documents

System searches related documents.

It may find:

Employee Handbook HR Policy Offer Letter Template Exit Process SOP

But it should select only the most relevant chunks.

Stage 3 — Send Context to LLM

The retrieved information is added to the prompt.

Example prompt:

Use the following context to answer the user question. Context: According to HR Policy 2026, employees must serve a 60-day notice period unless waived by management. Question: What is the notice period for resignation?

Stage 4 — LLM Answer

The LLM answers using the given context:

The notice period is 60 days unless it is waived by management.

So the LLM is not guessing. It is reading the retrieved context.

Important: Retrieval, context, and generation are separate responsibilities. If retrieval returns weak evidence, the final answer will also become weak.

14. Why Retrieval Comes Before Generation

This is very important.

Without retrieval:

LLM guesses from general knowledge.

With retrieval:

LLM first receives relevant facts, then answers.

Example:

Question:

What is the support SLA for Gold customers?

Without retrieval, the model may guess:

Gold customers receive 24-hour support.

But after retrieval, the actual document may say:

Gold customers receive response within 4 business hours.

So retrieval must happen before generation because:

The LLM needs the correct facts before writing the answer.

A simple analogy:

Normal LLM = student answering from memory RAG = student first opens the textbook, then answers
Memory hook: RAG is an open-book answer system. The book is searched before the student writes the answer.

15. What “Grounding” Means

Grounding means:

Connecting the LLM answer to real source material.

A grounded answer is based on retrieved documents.

Ungrounded answer:

I think the refund period is 30 days.

Grounded answer:

The refund period is 7 days, according to the Refund Policy, section 2.1.

Grounding reduces hallucination because the answer is tied to evidence.

In RAG, the retrieved document chunks act like evidence.

Retrieved chunks = evidence LLM answer = explanation based on evidence
Key idea: Grounding does not mean the answer is automatically perfect. It means the answer is connected to evidence that can be checked.

16. What “Context Injection” Means

Context injection means:

Putting retrieved information into the prompt before sending it to the LLM.

Example:

System: You are an HR assistant. Answer only using the provided context. Context: Employees are eligible for 18 annual leave days per year. User Question: How many annual leaves do employees get?

The retrieved context is injected into the prompt.

This does not permanently change the model.

It only gives the model temporary working information for that answer.

Important:

Context injection = temporary information inside the prompt Fine-tuning = longer-term change to model behavior
Practical note: Context injection is why RAG can use changing documents without retraining the model again and again.

17. What “Source-Based Answer” Means

A source-based answer means the answer shows where the information came from.

Example:

Employees receive 18 annual leave days per year. Source: HR Policy 2026, Section 4.1

This is important in enterprise systems because users should not blindly trust the answer.

They should know:

Which document was used? Which section was used? Is the source current? Can I verify it?

A good RAG answer should usually show:

Answer Source document name Page or section Date/version if available

This is one reason RAG is powerful for business.

Enterprise point: Source-based answers support trust, verification, audit, compliance, and debugging.

18. Why RAG Makes LLM More Useful

RAG makes LLMs useful because it connects them to real working knowledge.

Without RAG, an LLM is like:

A smart person with no access to your files.

With RAG, an LLM becomes:

A smart assistant who can search your files, read relevant parts, and explain them.

RAG helps with:

Private data Latest data Large document collections Source-based answers Lower hallucination Enterprise knowledge access Document Q&A Training assistants Support assistants Policy assistants

For example, for your AI learning website, RAG can later allow:

User asks: What did we learn in Transformer Architecture? System retrieves: Topic 11 notes, demo explanation, README section LLM answers: A clear explanation with references to the exact topic section.

That means your learning project can become an intelligent study assistant.

Project connection: This connects directly to your learning website. Later, a learner can ask questions from your AI/LLM notes and get answers with topic/step/source references.

Detailed Teaching Content

19. High-level RAG architecture

A RAG system has two big phases.

Phase 1 — Indexing phase Phase 2 — Query answering phase

Phase 1 — Indexing phase

This happens before the user asks a question.

Documents ↓ Clean text ↓ Split into chunks ↓ Create embeddings ↓ Store chunks + embeddings + metadata

Example:

You upload an HR policy PDF.

The system does this:

HR Policy PDF ↓ Extract text ↓ Split into sections ↓ Convert each section into embedding vector ↓ Store in vector database

This prepares the knowledge base.

Phase 2 — Query answering phase

This happens when the user asks a question.

User question ↓ Create query embedding ↓ Search vector database ↓ Retrieve matching chunks ↓ Build prompt with context ↓ Send to LLM ↓ Generate answer ↓ Show answer with sources

So RAG architecture is not just one LLM call.

It is a full pipeline.

Teaching note: The indexing phase prepares the library. The query answering phase uses that library to answer the user.

20. User query

The user query is the question asked by the user.

Example:

What is the notice period for resignation?

In a RAG system, this query is not immediately sent to the LLM.

First, it is used to search the knowledge base.

Important point:

In RAG, the user query has two jobs: 1. Search for relevant information 2. Ask the LLM to answer

A normal chatbot uses the query mainly for answering.

A RAG chatbot uses the query first for retrieval.

Mini check: Do not think of the user question as only a prompt. In RAG, it is also the search key that begins retrieval.

21. Query processing

Before searching, the system may clean or improve the query.

Example user query:

leave carry forward?

The system may rewrite it as:

What is the company policy for carrying forward unused leave?

Query processing may include:

Cleaning the text Removing unnecessary words Expanding abbreviations Rewriting unclear questions Detecting intent Adding filters Breaking a complex question into smaller questions

Example:

User asks:

Compare leave policy for interns and full-time employees.

This may be split into:

1. Retrieve leave policy for interns 2. Retrieve leave policy for full-time employees 3. Compare both

This becomes important in advanced RAG.

Teaching note: Query processing improves retrieval quality. If the query is unclear, the retriever may search the wrong thing.

22. Document store

The document store holds the original documents or extracted text.

Examples:

PDF files Word documents HTML pages Markdown files Database rows Emails Support tickets Policy documents Training notes

The document store is important because the vector database may store chunks, but the system still needs to know where the chunk came from.

Example metadata:

Document name: HR Policy 2026 Page: 12 Section: Leave Carry Forward Version: v3 Date: 2026-01-10 Department: HR

This helps the system show sources later.

Important: Without source information, a RAG answer may be hard to verify even if the text is correct.

23. Vector database

A vector database stores embeddings.

An embedding is a number representation of meaning.

Example:

"Employees get 12 casual leaves" ↓ [0.12, -0.44, 0.87, 0.05, ...]

The vector database stores:

Chunk text Embedding vector Metadata Source reference

When the user asks a question, the query is also converted into a vector.

Then the vector database searches for chunks with similar meaning.

Example:

User query:

How many casual leaves do I get?

It can match this chunk:

Employees are allowed 12 casual leaves per year.

Even though the exact words are different, the meaning is similar.

That is why vector search is powerful.

Simple memory: Vector database = smart meaning index for document chunks.

24. Embedding model

The embedding model converts text into vectors.

It is used two times in RAG.

During indexing

Document chunk → embedding vector

During question answering

User question → query embedding vector

The same or compatible embedding model should be used for both.

Why?

Because both document chunks and queries need to live in the same meaning space.

Simple analogy:

If documents are mapped using one coordinate system, the query should also use the same coordinate system.

Otherwise similarity search becomes unreliable.

Important: If you change the embedding model, you may need to re-embed your document chunks.

25. Retriever

The retriever is the part that finds useful chunks.

Input:

User question or query embedding

Output:

Top matching chunks

Example:

User asks:

Can unused leaves be carried forward?

Retriever returns:

Chunk 1: Annual leave carry-forward rules Chunk 2: Casual leave policy Chunk 3: Leave expiry rule

Retriever may use:

Vector search Keyword search Hybrid search Metadata filters Re-ranking

The retriever is one of the most important parts of RAG.

A weak retriever gives bad context.

Bad context leads to bad answers.

Wrong retrieval → wrong answer Good retrieval → better grounded answer
Key rule: The LLM can only answer from what the retriever gives it.

26. Context builder

The context builder prepares the final text that will be given to the LLM.

It takes retrieved chunks and decides:

Which chunks to include Which chunks to remove In what order to place them How much text can fit Whether to include source names Whether to compress or summarize chunks

Example retrieved chunks:

Chunk A — Leave policy Chunk B — Holiday calendar Chunk C — Leave carry-forward rule Chunk D — Payroll unrelated text

The context builder may keep only:

Chunk A Chunk C

Then build this:

Context: [Source: HR Policy 2026, Section 4.1] Employees receive 18 annual leave days per year. [Source: HR Policy 2026, Section 4.3] Only 5 unused annual leave days may be carried forward.

This final context goes into the prompt.

Teaching note: Retrieval finds possible material. Context building prepares usable evidence.

27. Prompt template

The prompt template tells the LLM how to answer.

A RAG prompt usually contains:

System instruction Retrieved context User question Answer rules Citation rules Fallback rule

Example:

You are a helpful HR assistant. Use only the provided context to answer. If the answer is not in the context, say: "I could not find this in the provided documents." Context: {retrieved_context} Question: {user_question} Answer with source references.

This is very important because the LLM should not freely guess.

The prompt controls how the retrieved content is used.

Important: Prompt template is where we tell the model not to hallucinate, how to cite, and how to respond when the answer is missing.

28. LLM

The LLM is the generator.

Its role is not to search the documents.

Its role is to read the retrieved context and produce a clear answer.

Input to LLM:

Prompt template + retrieved context + user question

Output from LLM:

Final answer

In RAG, the LLM should act like:

A reader and explainer of the retrieved evidence

Not like:

A person guessing from memory
Simple memory: The LLM is the writer, not the librarian.

29. Answer generation

Answer generation is where the LLM writes the final response.

A good RAG answer should be:

Relevant Grounded Clear Source-based Not overconfident Honest when information is missing

Example:

Question:

How many annual leaves can be carried forward?

Retrieved context:

Only 5 unused annual leave days may be carried forward to the next calendar year.

Generated answer:

Employees can carry forward up to 5 unused annual leave days to the next calendar year. Source: HR Policy 2026, Section 4.3

Bad RAG answer:

Usually companies allow 10 to 15 leaves to be carried forward.

Why bad?

Because it ignores the retrieved source and gives a general answer.

Key rule: A RAG answer should come from retrieved evidence, not general guessing.

30. Citation/source display

Citation means showing where the answer came from.

In enterprise RAG, this is very important.

A source display may include:

Document name Page number Section heading Date/version URL or file path Confidence score

Example:

Source: HR Policy 2026.pdf, Page 12, Section 4.3

Why citations matter:

Users can verify the answer Trust increases Mistakes can be traced Compliance becomes easier Auditing becomes possible

Without source display, the user has no easy way to check whether the answer is correct.

Enterprise point: A citation is not decoration. It is part of trust, compliance, and debugging.

31. Feedback loop

A real RAG system should improve over time.

User feedback can be:

Helpful / not helpful Correct / incorrect Missing source Wrong source Outdated answer Need more details

The system can log:

User question Retrieved chunks LLM answer Sources shown User feedback Correction if any

This helps improve:

Chunking Retrieval Prompting Metadata Document freshness Evaluation dataset

Example:

If users often say:

Answer is wrong because old policy was used.

Then we know we need better document versioning or freshness filtering.

Improvement rule: RAG is not build once and forget. It needs feedback, evaluation, and improvement.

The key point:

RAG is not only an LLM. RAG is a search + context + generation architecture.

Full Teaching Flow

Opening mental model

Let us take this slowly. In RAG, the LLM is not the first thing we should worry about. The first question is: what material is the LLM going to read before answering?

If the material is clean, meaningful, current, and properly split, then the RAG system has a strong foundation. If the material is noisy, broken, outdated, wrongly split, or missing source information, even a strong LLM will struggle.

Good source material ↓ Good extraction ↓ Good chunks ↓ Good embeddings ↓ Good retrieval ↓ Good context ↓ Better grounded answer

So Step 4 has two connected parts:

Part 4 — What kind of documents and knowledge can be used in RAG? Part 5 — How do we split those documents into chunks?

Think of RAG like a student answering from a textbook. The LLM is the student. The retrieved chunks are the textbook pages. Chunking decides which pages exist and how readable they are.

Part 4

Documents and Knowledge Base

32. What kind of data can be used in RAG

RAG can work with almost any information source, as long as we can extract useful text or structured content from it. This is why RAG is used in companies: the useful knowledge is scattered across many systems, not only in one document folder.

PDF documents Word documents Web pages Emails SQL database rows Logs Policies Product manuals Support tickets Training material Code repositories Enterprise knowledge bases

The important point is this: RAG does not understand a file just because the file exists. The file must be converted into searchable knowledge. A PDF must become text. A web page must become clean article content. An email thread must become useful messages. SQL rows must become either text chunks or live SQL queries.

Raw source ↓ Extract useful content ↓ Clean noise ↓ Preserve metadata ↓ Split into chunks ↓ Create embeddings ↓ Ready for retrieval

So when we say “RAG uses documents,” what we really mean is: RAG uses clean, chunked, searchable representations of those documents.

Teaching rule: A RAG system is only as strong as the knowledge base it is allowed to retrieve from.

33. PDFs

PDFs are very common in RAG. Companies keep HR policies, product manuals, SOPs, contracts, audit reports, training PDFs, brochures, and technical documentation as PDFs.

But PDFs can be tricky because they were designed for fixed visual layout, not always for clean text extraction.

A PDF may contain: Selectable text Scanned image text Tables Headers and footers Page numbers Two-column layouts Diagrams Watermarks Footnotes

If the PDF is scanned, we may need OCR. If the PDF has tables, normal text extraction may break table structure. If the PDF has two columns, the extraction may read the left column and right column in the wrong order.

Example of a good original PDF rule:

Leave Carry Forward: Employees may carry forward up to 5 unused annual leave days.

Bad extraction may turn it into something like this:

Employees carry forward annual Page 12 HR Policy 2026 5 unused leave days may

This may still look like text, but for RAG it is poor evidence. The retriever may retrieve it, but the LLM may not understand the full rule clearly.

For PDFs, we should inspect extraction quality before embedding. We should check whether headings, table rows, bullets, and page numbers are clean enough.

PDF rule: Do not assume the PDF was read correctly. In RAG, the extracted text is the real input, not the visual PDF.

34. Word documents

Word documents are often easier than PDFs because headings, paragraphs, bullets, and tables are more structured. HR policies, project documents, internal manuals, proposals, training notes, and SOPs may exist as Word files.

The useful thing about Word documents is that headings can become metadata. For example:

Document: HR Policy 2026 Heading path: Leave Policy → Carry Forward Text: Employees may carry forward up to 5 unused annual leave days.

This heading path is very useful because even if the text is short, the heading tells the retriever and the LLM what the chunk is about.

Word documents may also contain comments, track changes, tables, embedded images, and old revision content. In enterprise RAG, we must decide what should be indexed and what should be ignored.

Word document rule: Preserve headings and section hierarchy. They improve chunk meaning, retrieval, and citations.

35. Web pages

RAG can also use web pages. This may include public documentation, internal portals, product help pages, API docs, blog articles, and knowledge-base pages.

But web pages include more than useful content. They often contain noise.

Navigation menus Cookie banners Footer links Ads Related article links Comments Repeated headers Sidebars

If we index the raw page without cleaning, the vector database may store the same menu text hundreds of times. This creates retrieval noise.

Example: the useful content may be:

Refunds are allowed within 7 days of purchase.

But the extracted page may also contain:

Home | Pricing | Login | Contact | Terms | Newsletter | Related posts

Good extraction keeps the main content and removes layout noise. For web-page RAG, we should also store the URL, title, page date, and section heading so that answers can cite the exact page.

Web page rule: Extract the article or documentation body, not the whole visual page.

36. Emails

Emails are useful in enterprise RAG, especially for internal knowledge. Many important decisions are buried inside email threads: client approvals, project clarifications, vendor commitments, escalation replies, and meeting follow-ups.

Example email sources:

Client communication Project decisions Approval messages Support escalations Meeting follow-ups Vendor communication Attachment references

But emails are sensitive and messy. A single thread may contain repeated replies, signatures, disclaimers, forwarded content, and confidential information.

Possible email noise: Long threads Repeated replies Signatures Legal disclaimers Old attachments Confidential information PII such as phone numbers and addresses

Example user question:

What did the client approve last week?

A RAG system may retrieve the correct email thread and summarize the approval. But only authorized users should see that information.

So for email RAG, access control and cleaning are critical. We should preserve sender, date, subject, thread, and attachment metadata, but avoid leaking private content to unauthorized users.

Email rule: Emails can be powerful RAG sources, but they need strict permissions, thread cleaning, and privacy handling.

37. SQL data

RAG can work with SQL data in two ways.

Way 1 — Convert rows into text chunks

Example database row:

Product: Premium Support SLA: 4 business hours Region: India Effective date: 2026-01-01

Converted into text:

Premium Support has a 4 business hour SLA in India. This rule is effective from 2026-01-01.

Then the text can be embedded and stored in a vector database.

Way 2 — Use SQL as a tool

Instead of embedding all rows, the system can ask the database directly. This is better for exact counts, live values, filters, and aggregations.

Example:

Question: How many orders were placed yesterday? Tool: Run SQL query Answer: Yesterday, 1,248 orders were placed.

This second approach is closer to tool calling or SQL agent behavior, but it can be combined with RAG.

For example, RAG may retrieve a policy explanation while SQL provides the current number.

RAG retrieves: what the SLA rule means SQL retrieves: how many tickets violated the SLA this week
SQL rule: Use RAG for explanation from text. Use SQL tools for exact live data, counts, filters, and calculations.

38. Logs

Logs can also be used in RAG, especially in IT operations, DevOps, observability, and support systems.

Example logs:

Application logs Server logs Deployment logs Security logs API logs Database logs Job-processing logs

A user may ask:

What errors happened after yesterday’s deployment? Which API failed most frequently? Show me payment gateway timeout patterns.

But logs are huge, repetitive, and noisy. If we blindly embed all logs, retrieval becomes expensive and messy.

So for logs, we usually need filtering and summarization before RAG.

Filter by time window Filter by service Group repeated errors Summarize patterns Store incident-level chunks Attach timestamp and severity metadata

For log RAG, raw logs are often less useful than cleaned summaries of incidents, error patterns, and root-cause notes.

Logs rule: Do not treat every log line as a normal document chunk. Filter, group, summarize, and preserve time metadata.

39. Policies

Policies are one of the best RAG use cases because people naturally ask questions about rules and benefits.

Example policies:

HR policy Security policy Travel policy Expense policy Leave policy Data privacy policy Work-from-home policy

Users may ask natural questions:

Can I carry forward unused leave? What is the reimbursement limit for hotel stay? Can contractors access production data?

RAG can retrieve the exact policy section and give an answer with source.

This is better than asking users to manually search a 100-page policy PDF.

For policy RAG, version is very important. A 2024 policy and a 2026 policy may say different things.

Policy rule: Store policy name, section, version, effective date, department, region, and source page.

40. Product manuals

Product manuals are another strong use case. They help support teams, customers, field engineers, and training teams answer how-to and troubleshooting questions.

Example manual content:

Software manuals Hardware manuals Installation guides Troubleshooting guides API manuals Configuration guides

User asks:

How do I reset the device? What does error code PX-104 mean? How do I configure SSO?

RAG retrieves the exact manual section and generates a step-by-step answer.

In product manuals, exact terms matter. Error codes, version numbers, model numbers, and configuration names should be preserved.

Manual rule: Use metadata such as product version, model number, operating system, page number, and section name.

41. Support tickets

Support tickets contain practical knowledge because they record real problems and real resolutions.

Example ticket content:

Past customer issues Known bugs Workarounds Fix history Resolution notes Escalation steps

User asks:

Has this error happened before? What was the previous solution? Which workaround solved this issue last time?

RAG can retrieve similar past tickets and suggest how the issue was resolved.

But support tickets also contain customer names, confidential details, logs, internal comments, and repeated conversation noise.

So ticket RAG needs cleaning, PII removal, and permission checks. The system should also distinguish between verified fixes and casual comments.

Ticket rule: Past tickets are useful, but do not blindly treat every ticket comment as trusted product truth.

42. Training material

Training material is especially important for our learning project. Your AI/LLM learning site can later become a RAG-powered study assistant.

Training material can include:

Course notes Markdown files HTML pages PDF handouts Code examples Diagrams Quiz questions Answer keys Demo explanations

For your AI/LLM website, RAG can later let a learner ask:

Explain attention again from our notes. Where did we cover cosine similarity? What is the difference between RAG and embeddings?

Then the system retrieves the matching topic section and answers from your own material.

This can become your own AI learning assistant. It can cite topic number, step number, section name, and demo file.

Training rule: Preserve topic, step, heading, quiz, demo, and README metadata so the answer can point learners back to the exact lesson.

43. Code repositories

RAG can also work with code repositories. This is useful for developer assistants, onboarding tools, architecture Q&A, and codebase documentation.

Example repository sources:

README files Source code API controllers Configuration files Architecture docs Comments Test cases

Developer questions:

Where is authentication handled? Which API updates order status? How does the retry policy work? What does this function do?

Code RAG requires special care because code should not be split randomly.

For example, splitting a function in the middle may reduce understanding. A better approach is to chunk by file, class, method, module, or documentation block.

Code rule: Preserve file path, language, class name, function name, and repository branch when creating chunks.

44. Enterprise knowledge base

An enterprise knowledge base may combine many source systems together.

SharePoint documents Confluence pages Google Drive files ServiceNow tickets Jira issues GitHub repositories Email threads SQL databases Internal wikis Training portals

Enterprise RAG is powerful, but difficult because it needs:

Permissions Freshness Security Version control Audit logs Source traceability Cost control Monitoring

This is why enterprise RAG is more than just a vector database demo. It is a full knowledge system with governance.

In an enterprise setup, the same question may return different results for different users because they have different permissions.

Enterprise rule: Apply access control before retrieval. Restricted chunks should not enter the prompt at all.
Part 5

Chunking

45. What is chunking

Chunking means splitting a large document into smaller pieces. Instead of storing and retrieving an entire 300-page PDF, we store smaller meaningful chunks.

Example document:

HR Policy 2026 - Leave Policy - Work From Home Policy - Travel Policy - Expense Policy

Instead of storing the whole document as one piece, we split it into chunks:

Chunk 1: Leave Policy Chunk 2: Work From Home Policy Chunk 3: Travel Policy Chunk 4: Expense Policy

Why? Because when the user asks about leave carry-forward, we only want to retrieve the leave-related part, not the whole HR policy.

Chunking is the bridge between documents and retrieval. If chunks are meaningful, retrieval becomes easier. If chunks are broken, retrieval becomes weak.

Chunking rule: Split documents into meaningful pieces, not random fragments.

46. Why we cannot send full documents to the LLM

There are several reasons we cannot simply send the full document to the LLM every time.

1. Context window limit 2. Cost 3. Latency 4. Noise 5. Lower answer quality

If a PDF has 300 pages and the user asks one question about leave policy, sending travel policy, security policy, and payroll rules creates noise.

RAG should send only the useful parts.

Good RAG = small relevant context Bad RAG = large noisy context

Also, the LLM may miss the answer if the useful sentence is buried in a huge context. More context is not always better.

Context rule: The goal is not maximum text. The goal is enough relevant evidence.

47. Chunk size

Chunk size means how big each chunk is.

Example:

Small chunk: 100 words Medium chunk: 300–600 words Large chunk: 1000+ words

There is no universal perfect size. It depends on:

Document type Question type Embedding model LLM context window Need for citations Domain complexity

A chunk should be large enough to contain meaning, but small enough to avoid too much unrelated noise.

Example: a chunk that only says “up to 5 days” is too small. A chunk that includes all leave policy, payroll policy, travel policy, and expense policy may be too large.

Chunk-size rule: Use enough text to preserve meaning, but not so much that retrieval becomes noisy.

48. Chunk overlap

Chunk overlap means repeating a small part from the previous chunk into the next chunk.

Example without overlap:

Chunk 1: Employees may carry forward unused annual leave Chunk 2: up to 5 days into the next calendar year.

Individually, both chunks are incomplete. The first chunk does not say how many days. The second chunk does not say what rule it belongs to.

With overlap, the meaning is preserved:

Chunk 1: Employees may carry forward unused annual leave up to 5 days. Chunk 2: Unused annual leave up to 5 days may be carried into the next calendar year.

Overlap is useful when important meaning sits at the boundary between two chunks.

But too much overlap creates duplicate chunks and wastes storage/context.

Overlap rule: Use overlap to protect boundary meaning, but avoid excessive duplication.

49. Small chunks vs large chunks

Small chunks and large chunks each have advantages and problems.

Chunk TypeAdvantageProblem
Small chunksMore focused retrievalMay lose meaning
Large chunksMore complete contextMay retrieve irrelevant text
Medium chunksBalancedUsually best starting point

Example too-small chunk:

up to 5 days.

This lacks meaning. The LLM may ask: 5 days of what?

Example too-large chunk:

The full HR policy including leave, holidays, payroll, travel, resignation, expenses, and security rules...

This has too much noise. Retrieval may be less precise.

Good chunk:

Leave Carry Forward: Employees may carry forward up to 5 unused annual leave days into the next calendar year.
Practical rule: Start with meaningful section-sized chunks, then evaluate retrieval quality.

50. Semantic chunking

Semantic chunking means splitting based on meaning, not fixed size only.

Instead of blindly cutting every 500 words, we split by natural boundaries.

Example semantic boundaries:

Heading Paragraph FAQ question-answer pair Table row group Function or class in code Policy subsection

For policy documents, semantic chunks may be section-wise. For code, semantic chunks may be function-wise. For FAQs, each question-answer pair can be one chunk.

Semantic chunking usually improves retrieval quality because each chunk contains one coherent idea.

Semantic rule: Let document meaning guide the split, not only character count.

51. Page-based chunking

Page-based chunking splits by page.

Example:

Page 1 → Chunk 1 Page 2 → Chunk 2 Page 3 → Chunk 3

This is simple and useful when page citations matter.

But page-based chunking can be weak if a topic continues across pages.

Example:

Page 10 ends with: Employees may carry forward unused annual leave Page 11 begins with: up to 5 days into the next calendar year.

The meaning is split across two pages. So page-based chunking is easy, but not always best.

Page rule: Page numbers help citation, but chunking should still preserve full meaning.

52. Paragraph-based chunking

Paragraph-based chunking splits by paragraphs. This often works better than page-based chunking because paragraphs usually contain one idea.

Example:

Paragraph 1: Annual leave eligibility Paragraph 2: Leave carry-forward Paragraph 3: Leave encashment

Each paragraph becomes one chunk or part of a chunk.

Problem: some paragraphs are too small. A paragraph like “Eligibility” or “Conditions apply” may not be meaningful alone.

So systems often combine paragraphs until they reach a useful chunk size.

Paragraph rule: Paragraph boundaries are useful, but very small paragraphs may need to be merged.

53. Heading-aware chunking

Heading-aware chunking uses document headings to decide chunk boundaries and metadata.

Example document structure:

4. Leave Policy 4.1 Annual Leave 4.2 Casual Leave 4.3 Carry Forward 4.4 Encashment

A good chunk may include the heading path:

HR Policy 2026 > Leave Policy > Carry Forward Employees may carry forward up to 5 unused annual leave days.

This is very powerful because the heading gives context even if the chunk text is short.

Heading-aware chunking also improves citations. Instead of citing only a file, we can cite the exact section.

Heading rule: Preserve the heading path. It improves retrieval, generation, and source display.

54. Sliding window chunking

Sliding window chunking creates overlapping chunks by moving a window through the text.

Example:

Words 1–100 → Chunk 1 Words 51–150 → Chunk 2 Words 101–200 → Chunk 3

Here the overlap is 50 words.

This is useful when we do not have good headings or structure.

But it can create duplicate chunks and repeated retrieval results.

So later we may need deduplication or re-ranking.

Sliding-window rule: Useful for unstructured text, but watch out for duplicate context.

55. Chunk metadata

Metadata means extra information stored with each chunk.

Example metadata:

Document name Page number Section heading Author Department Version Date File path URL Security level Language Created date Updated date

Chunk text:

Employees may carry forward up to 5 unused annual leave days.

Metadata:

Document: HR Policy 2026 Section: Leave Carry Forward Page: 12 Department: HR Version: v1.0 Effective date: 2026-01-01

Metadata is very important for:

Filtering Citations Security Freshness Debugging Audit

Example user question:

What is the 2026 leave policy?

Metadata can filter only documents where:

Year = 2026 Department = HR Document type = Policy
Metadata rule: Without metadata, RAG becomes difficult to filter, cite, secure, debug, and audit.

56. Bad chunking problems

Bad chunking causes bad RAG.

Common problems:

Chunk is too small Chunk is too large Chunk cuts a sentence Chunk loses heading Chunk mixes unrelated topics Chunk has repeated footer Chunk has broken table Chunk splits code badly Chunk has no metadata Chunk has outdated text

Example bad chunk:

5 days. Employees must apply before March.

We do not know 5 days of what. Leave? Refund? Notice? SLA?

Better chunk:

Leave Carry Forward: Employees may carry forward up to 5 unused annual leave days. They must apply before March.

Bad chunking creates weak embeddings. Weak embeddings create weak retrieval. Weak retrieval creates weak answers.

Bad-chunking rule: If the chunk does not make sense to a human, it may not make sense to the RAG system either.

57. How chunking affects retrieval quality

Chunking directly affects retrieval quality.

Good chunking means:

Relevant chunks are retrieved Chunks contain enough meaning Metadata supports filtering Answer quality improves Hallucination reduces

Bad chunking means:

Retriever misses correct information Retriever finds incomplete information LLM receives confusing context Answer becomes vague or wrong Sources become hard to verify

Important rule:

RAG quality starts before retrieval. It starts with document preparation and chunking.

If we improve chunking, retrieval often improves even without changing the LLM.

For production RAG, chunking should be tested with real user questions. We should ask: did the correct chunk appear in the top results?

Final Step 4 rule: Good chunking creates better evidence. Better evidence creates better grounded answers.

Part 6 — Embeddings in RAG

58. Why embeddings are used in RAG

RAG needs to find information by meaning, not only by exact words.

Example user question:

Can I take leave next month?

Relevant document chunk:

Employees may apply for annual vacation days subject to manager approval.

The words are different:

leave ≠ vacation
take ≠ apply

But the meaning is related.

Embeddings help with this.

An embedding converts text into a list of numbers that represents meaning.

Text → Embedding vector

Example:

"annual leave policy"
        ↓
[0.21, -0.08, 0.72, 0.14, ...]

In RAG, embeddings allow us to search by semantic similarity.

59. Document embeddings

When we prepare documents for RAG, each chunk is converted into an embedding.

Example chunk:

Employees may carry forward up to 5 unused annual leave days.

Embedding model converts it into:

[0.18, -0.32, 0.67, 0.09, ...]

The system stores:

Chunk text
Chunk embedding
Document name
Page number
Section
Metadata

So document embeddings are created during the indexing phase.

Document chunk → embedding → store in vector database

60. Query embeddings

When the user asks a question, the question is also converted into an embedding.

User question:

How many leaves can I carry forward?

Query embedding:

[0.20, -0.29, 0.71, 0.11, ...]

Then the system compares this query embedding with stored document embeddings.

If the query vector is close to a document chunk vector, that chunk is likely relevant.

User question → query embedding → compare with document embeddings

61. Meaning-based search

Meaning-based search is different from keyword search.

Keyword search looks for exact or near-exact words.

Meaning-based search looks for similar ideas.

Example:

User asks:

What is the notice period?

Document says:

Employees must serve 60 days before separation.

Keyword search may fail because the document does not contain the exact phrase “notice period”.

Embedding search may still find it because:

notice period ≈ serve 60 days before separation

That is why embeddings are central to RAG.

62. Similarity search

Similarity search means finding the nearest chunks to the query in embedding space.

Imagine each chunk is a point on a map.

Chunk A: Leave policy
Chunk B: Travel policy
Chunk C: Password reset
Chunk D: Resignation policy

If the user asks about resignation notice period, the query point should land near resignation-related chunks.

The retriever then returns the closest chunks.

Query embedding
      ↓
Find nearest document embeddings
      ↓
Return top matching chunks

63. Cosine similarity intuition

Cosine similarity measures how similar two vectors are by checking their direction.

Simple intuition:

Same direction = similar meaning
Different direction = different meaning
Opposite direction = unrelated or opposite meaning

Example:

Query:
How many casual leaves are allowed?

Chunk A:
Employees receive 12 casual leaves per year.

Chunk B:
Employees must reset passwords every 90 days.

Chunk A will have higher similarity.

Chunk B will have lower similarity.

Cosine similarity usually gives a score.

Chunk A similarity: 0.89
Chunk B similarity: 0.21

So the retriever chooses Chunk A.

68. Embedding dimension

Embedding dimension means the number of values in the vector.

Example:

384 dimensions
768 dimensions
1536 dimensions
3072 dimensions

Higher dimension may capture more information, but it can also mean:

More storage
More compute
Higher indexing cost
More memory usage

Important:

Bigger dimension does not always mean better RAG.
Quality depends on the model and use case.

69. Embedding refresh

When documents change, embeddings may need to be refreshed.

Example:

Old document chunk:

New document chunk:

If we do not update embeddings, the RAG system may retrieve old information.

Embedding refresh means:

Detect changed documents
Remove old chunks
Create new chunks
Generate new embeddings
Update vector database

This is very important in enterprise RAG.

70. Why embedding quality matters

Embedding quality affects what the retriever finds.

Good embeddings:

Find relevant chunks even with different wording
Handle synonyms
Understand domain concepts
Improve answer grounding
Reduce missed retrieval

Poor embeddings:

Retrieve irrelevant chunks
Miss correct chunks
Confuse similar but different topics
Lower answer quality
Increase hallucination risk

Important RAG rule:

The LLM can only answer well if the retriever gives it good context.
The retriever depends heavily on embeddings.

Part 7 — Vector Databases

71. What is a vector database

A vector database stores and searches embedding vectors.

It stores:

Text chunks
Embedding vectors
Metadata
Source information

Example stored record:

Chunk:
Employees may carry forward up to 5 unused annual leave days.

Embedding:
[0.18, -0.32, 0.67, ...]

Metadata:
Document = HR Policy 2026
Page = 12
Section = Leave Carry Forward
Department = HR

When a query comes, the vector database finds chunks with similar vectors.

72. Why normal SQL search is not enough

SQL is excellent for structured data.

Example:

SELECT * FROM Employees WHERE Department = 'HR';

But SQL keyword search is not enough for meaning-based retrieval.

Example question:

Can I leave the company after resigning?

Relevant document:

Employees must serve a 60-day notice period before separation.

A simple SQL LIKE '%leave%' search may retrieve wrong results about annual leave.

Vector search can understand the semantic meaning better.

However, SQL is still useful for:

Metadata filters
Structured facts
Permissions
Exact values
Reports
Transactions

Modern RAG often combines SQL + vector search.

73. Vector index

A vector index is a data structure that helps search vectors quickly.

Without an index, the system may compare the query vector with every stored vector.

That is slow for large datasets.

Example:

10 documents → direct comparison is fine
10 million chunks → direct comparison is too slow

Vector index helps find nearest vectors faster.

It is like an index at the back of a book, but for meaning.

74. Similarity search

In vector DB, similarity search means:

Find vectors closest to the query vector.

The result usually includes:

Chunk text
Similarity score
Metadata
Source

Example:

Query:
How many leaves can I carry forward?

Result 1:
Leave Carry Forward chunk
Score: 0.91

Result 2:
Annual Leave Eligibility chunk
Score: 0.79

Result 3:
Leave Encashment chunk
Score: 0.65

The RAG system may choose top results and pass them to the LLM.

75. Top-k retrieval

Top-k means how many chunks we retrieve.

Example:

Top-k = 3

The retriever returns the top 3 most similar chunks.

Choosing top-k is important.

Small top-k:

Less noise
Lower cost
But may miss needed context

Large top-k:

More context
May improve recall
But may add noise and cost

Example:

Top-k = 1 → only best chunk
Top-k = 5 → five chunks
Top-k = 20 → many chunks, possibly noisy

Good RAG systems tune top-k based on use case.

76. Approximate nearest neighbor search

Approximate nearest neighbor search, often called ANN, is used to search huge vector collections quickly.

Exact search:

Compare query with every vector.
Very accurate but slow for large data.

Approximate search:

Search smartly using an index.
Very fast, usually accurate enough.

In production, vector databases usually use ANN because datasets can be huge.

Trade-off:

Speed ↑
Scalability ↑
May miss a few exact nearest matches

But for most RAG systems, ANN works well.

77. Metadata filtering

Department = HR

The system can filter:

This improves:

Metadata filtering is essential in enterprise RAG.

78. Hybrid search

Hybrid search combines keyword search and vector search.

Why?

Because vector search is good for meaning, but keyword search is good for exact terms.

Example:

Embedding search may not handle this exact code well.

Keyword search can find it exactly.

Hybrid search combines both:

Keyword score + vector similarity score

Useful for:

Product codes
Policy IDs
Error codes
Part numbers
Legal clauses
Names
Exact phrases
Technical terms

In real enterprise RAG, hybrid search is often better than vector-only search.

89. What retrieval means

Retrieval means:

Finding the most relevant information for the user’s question.

In RAG, retrieval happens before the LLM answers.

Example:

User asks:

Can I carry forward unused leave?

The retriever searches the knowledge base and finds chunks like:

Chunk 1:
Employees may carry forward up to 5 unused annual leave days.

Chunk 2:
Unused casual leave cannot be carried forward.

Chunk 3:
Leave carry-forward must be approved before March.

Then these chunks are passed to the LLM as context.

Important idea:

Retriever decides what the LLM gets to read.

So retrieval quality strongly controls answer quality.

90. Keyword search

Keyword search finds documents using exact words or close word matches.

Example:

User asks:

leave carry forward

Keyword search looks for words like:

leave
carry
forward

It works well when the document uses the same words as the user.

Example document:

Employees may carry forward unused annual leave.

This will match well.

But keyword search can fail when the words are different.

User asks:

Can I take vacation days to next year?

Document says:

Employees may carry forward unused annual leave.

The meaning is similar, but the keywords are different.

That is where vector search helps.

91. Vector search

Vector search finds information by meaning.

It converts both the user question and document chunks into embeddings.

User question → query vector
Document chunks → document vectors

Then it finds the nearest document vectors.

Example:

User asks:

Can I take vacation days to next year?

Vector search can match:

Employees may carry forward unused annual leave.

Even though the words are different.

Vector search is strong for:

Synonyms
Natural language questions
Conceptual search
Similar meaning
Document Q&A

But vector search can be weak for exact codes, IDs, names, or numbers.

92. Hybrid retrieval

Hybrid retrieval combines keyword search and vector search.

Why?

Because both have strengths.

Retrieval TypeStrong AtWeak At
Keyword searchExact terms, IDs, error codes, namesSynonyms and meaning
Vector searchMeaning and natural languageExact values and codes
Hybrid searchCombines bothNeeds tuning

Example:

User asks:

What does error code PX-104 mean?

Keyword search is important because PX-104 is exact.

Example:

User asks:

How do I fix login timeout?

Vector search may help find chunks about session expiry, authentication delay, or token timeout.

Hybrid retrieval uses both.

Final retrieval score = keyword relevance + vector similarity

Enterprise RAG often uses hybrid retrieval.

93. Top-k documents

Top-k means how many chunks/documents we retrieve.

Example:

Top-k = 3

The retriever returns the top 3 most relevant chunks.

Example result:

1. Leave Carry Forward — score 0.91
2. Annual Leave Policy — score 0.82
3. Leave Approval Process — score 0.74

Top-k affects answer quality.

Small top-k:

Cleaner context
Lower cost
Faster answer
But may miss supporting details

Large top-k:

More coverage
Better chance of finding answer
But more noise
Higher cost
Slower answer

Simple practical starting point:

Top-k = 3 to 5 for small demos
Top-k = 5 to 10 for larger enterprise docs

But the correct value depends on document type and question complexity.

94. Similarity score

Similarity score tells how close the query is to a chunk.

Example:

Question:
How many annual leaves can be carried forward?

Chunk A:
Employees may carry forward up to 5 unused annual leave days.
Score: 0.92

Chunk B:
Employees are paid salary on the last working day.
Score: 0.18

Higher score usually means more relevant.

But we should not blindly trust scores.

Why?

Because sometimes:

A high score chunk may still not answer the question.
A lower score chunk may contain important supporting context.
Different embedding models produce different score ranges.
Scores are not always comparable across systems.

So similarity score is useful, but not the full truth.

95. Metadata filters

Metadata filters restrict retrieval to specific categories.

Example metadata:

Department = HR
Year = 2026
Region = India
DocumentType = Policy
SecurityLevel = Internal

User asks:

What is the India travel reimbursement policy for 2026?

The retriever can apply filters:

Region = India
Year = 2026
DocumentType = Travel Policy

Then it searches only inside matching chunks.

Metadata filters are important for:

Accuracy
Freshness
Security
Permissions
Cost reduction
Speed

Example security case:

A contractor should not retrieve confidential HR salary policy.

So metadata can include:

AllowedRoles = HR_Manager
SecurityLevel = Confidential

Retrieval must respect these filters.

96. Query expansion

Query expansion means adding related words or concepts to improve search.

User asks:

WFH rules

Expanded query:

work from home policy, remote work rules, hybrid work guidelines

Why useful?

Because documents may use different wording.

Original user term:

WFH

Document term:

remote work

Query expansion helps connect them.

It can be done using:

Synonym dictionary
Domain glossary
LLM-generated expansions
Past search logs
User profile/context

But it must be controlled.

Bad expansion can bring irrelevant results.

97. Query rewriting

Query rewriting means converting the user’s raw question into a clearer search query.

Example raw query:

carry forward?

Rewritten query:

What is the policy for carrying forward unused annual leave?

Another example:

User asks in conversation:

What about for interns?

This is incomplete.

If previous question was about leave policy, rewrite as:

What is the leave policy for interns?

Query rewriting is especially useful in chat because users often ask follow-up questions.

User: What is the leave policy?
User: What about interns?

The second question needs conversation context.

98. Multi-query retrieval

Multi-query retrieval creates multiple search queries from one user question.

Example:

User asks:

Compare leave rules for full-time employees and interns.

The system creates:

Query 1: full-time employee leave rules
Query 2: intern leave rules
Query 3: leave eligibility comparison

Then it retrieves chunks for each query and combines them.

This is useful when one question has multiple parts.

Examples:

Compare A and B
Find pros and cons
Summarize changes between versions
Explain policy for two roles
Answer multi-hop questions

Multi-query retrieval improves recall, but it can increase cost and noise.

99. Parent-child retrieval

Parent-child retrieval solves a common chunking problem.

Sometimes small chunks are good for search, but too small for answering.

So we store:

Child chunks = small searchable chunks
Parent chunks = larger context sections

Flow:

Search child chunks
      ↓
Find best child match
      ↓
Return its larger parent section to the LLM

Example:

Child chunk:

up to 5 unused annual leave days

Parent section:

Leave Carry Forward:
Employees may carry forward up to 5 unused annual leave days into the next calendar year. Carry-forward requires manager approval before March.

This gives both precision and context.

100. Self-query retrieval

Self-query retrieval means the system understands the user question and automatically creates both:

Search text
Metadata filters

Example user asks:

Show me HR policies from 2026 about leave in India.

The system extracts:

Search text: leave policy
Filters:
Department = HR
Year = 2026
Region = India

Then it retrieves using both meaning and metadata.

This is useful when users naturally include filters in their questions.

101. Recursive retrieval

Recursive retrieval means retrieval happens in multiple stages.

Example:

User asks:

What is the complete onboarding process for a new vendor?

The system may first retrieve a high-level onboarding document.

That document mentions:

Compliance checklist
Payment setup
Contract approval
Security review

Then the system retrieves more details for each of those sections.

Flow:

Retrieve overview
      ↓
Identify subtopics
      ↓
Retrieve details for each subtopic
      ↓
Combine answer

Recursive retrieval is useful for complex questions where one retrieval pass is not enough.

102. Retrieval failure examples

Retrieval can fail in many ways.

Failure 1 — Wrong document retrieved

Question:

What is the leave carry-forward rule?

Retriever returns travel policy.

Result: LLM answers wrongly or says irrelevant things.

Failure 2 — Correct document not retrieved

The answer exists, but retriever misses it.

Possible reasons:

Bad embeddings
Bad chunking
Wrong top-k
No query rewriting
Poor metadata

Failure 3 — Too much noisy context

Retriever returns 15 chunks, but only 1 is useful.

The LLM gets confused.

More context is not always better.
Relevant context is better.

Failure 4 — Old document retrieved

User asks current policy.

Retriever returns 2023 policy instead of 2026 policy.

Solution:

Use metadata filters
Versioning
Freshness ranking
Remove outdated chunks

Failure 5 — Security failure

User retrieves a document they should not access.

This is serious.

Solution:

Permission-aware retrieval
Document-level security
Role-based filters
Audit logs

Retrieval is the part that decides which information reaches the LLM.

Keyword search finds exact words.
Vector search finds meaning.
Hybrid retrieval combines both.
Top-k controls how many chunks are retrieved.
Similarity score ranks chunks.
Metadata filters improve accuracy and security.
Query expansion and rewriting improve search quality.
Multi-query, parent-child, self-query, and recursive retrieval handle advanced cases.

The key rule:

Good retrieval gives the LLM good evidence.
Bad retrieval gives the LLM confusion.

103. What happens after retrieval

Retrieval gives us some matching chunks.

Example user question:

Can I carry forward unused annual leave?

Retriever may return:

Chunk 1 — Leave Carry Forward
Chunk 2 — Annual Leave Eligibility
Chunk 3 — Leave Approval Process
Chunk 4 — Holiday Calendar
Chunk 5 — Payroll Rules

But not all retrieved chunks are equally useful.

After retrieval, the system must prepare the final context for the LLM.

This step is called context building.

Retrieval finds possible evidence.
Context building prepares useful evidence.

104. Selecting useful chunks

The context builder decides which retrieved chunks should go into the final prompt.

It should keep chunks that directly help answer the question.

For example:

Question:

Can unused annual leave be carried forward?

Useful chunks:

Leave carry-forward rule
Annual leave definition
Approval deadline

Less useful chunks:

Holiday calendar
Payroll date
Office parking policy

The goal is:

Keep relevant context.
Remove noise.

Because noisy chunks can confuse the LLM.

105. Ranking retrieved chunks

Retrieved chunks usually come with scores.

Example:

Chunk A — Leave Carry Forward — 0.91
Chunk B — Annual Leave Policy — 0.82
Chunk C — Leave Encashment — 0.70
Chunk D — Holiday List — 0.42

The context builder may rank chunks using:

Similarity score
Keyword match
Metadata match
Freshness
Document authority
User permissions
Re-ranker score

Example:

A newer policy document should rank above an old one.

HR Policy 2026 should rank above HR Policy 2023

Ranking is important because the LLM will depend heavily on the final context order.

106. Removing duplicates

RAG systems often retrieve duplicate or near-duplicate chunks.

Example:

Chunk 1:
Employees may carry forward up to 5 unused annual leave days.

Chunk 2:
Annual leave carry-forward is limited to 5 unused days.

Chunk 3:
Employees may carry forward up to 5 unused annual leave days.

If we send all three, we waste context space.

The context builder should remove duplicates.

Why?

Save context window space
Reduce repetition
Reduce cost
Improve clarity
Avoid overemphasizing duplicate information

Deduplication becomes important when documents have repeated content, copied policies, headers, footers, or versioned documents.

107. Combining chunks

Sometimes one chunk alone is not enough.

Example:

Question:

What is the leave carry-forward rule and deadline?

Retrieved chunks:

Chunk 1:
Employees may carry forward up to 5 unused annual leave days.

Chunk 2:
Carry-forward request must be submitted before March 31.

The context builder combines both.

Final context:

Employees may carry forward up to 5 unused annual leave days.
Carry-forward request must be submitted before March 31.

Combining chunks helps answer multi-part questions.

But we should combine only related chunks.

Bad combination:

Leave carry-forward + travel reimbursement + parking rules

This creates confusion.

108. Respecting context window limit

The LLM has a context window limit.

That means it can only read a certain amount of text at once.

The prompt includes:

System instruction
Retrieved context
User question
Conversation history
Output instructions

So we cannot fill everything with document chunks.

The context builder must decide:

How many chunks can fit?
How much text from each chunk?
Which chunks are most important?
Should older chat history be summarized?
Should context be compressed?

Important rule:

RAG is not about sending maximum text.
RAG is about sending the best useful text.

109. Adding metadata

Metadata gives meaning around the chunk.

Chunk text alone:

Employees may carry forward up to 5 unused annual leave days.

With metadata:

Document: HR Policy 2026
Section: Leave Policy > Carry Forward
Page: 12
Version: v3
Effective Date: 2026-01-01
Department: HR

Metadata helps the LLM and the user understand:

Where the chunk came from
Whether it is current
Which department owns it
Whether it is trustworthy
How to cite it

Metadata is especially useful when there are conflicting documents.

Example:

HR Policy 2023 says 10 days.
HR Policy 2026 says 5 days.

Metadata helps choose the latest and correct source.

110. Adding source names

Source names are used for citations.

Example final answer:

Employees may carry forward up to 5 unused annual leave days.

Source: HR Policy 2026, Section 4.3

To make this possible, the context should include source labels.

Example final context:

[Source: HR Policy 2026, Section 4.3, Page 12]
Employees may carry forward up to 5 unused annual leave days.

[Source: HR Policy 2026, Section 4.4, Page 13]
Carry-forward requests must be submitted before March 31.

This makes the LLM’s job easier.

Without source labels, the LLM may answer correctly but not cite properly.

111. Creating final context

Final context is the exact content passed to the LLM.

It may look like this:

Context:
[Source 1: HR Policy 2026 > Leave Policy > Carry Forward, Page 12]
Employees may carry forward up to 5 unused annual leave days.

[Source 2: HR Policy 2026 > Leave Policy > Deadline, Page 13]
Carry-forward requests must be submitted before March 31.

Question:
Can I carry forward unused annual leave, and is there a deadline?

The LLM then generates:

Yes. Employees may carry forward up to 5 unused annual leave days. 
The request must be submitted before March 31.

Sources:
- HR Policy 2026, Leave Policy > Carry Forward, Page 12
- HR Policy 2026, Leave Policy > Deadline, Page 13

So final context is the bridge between retrieval and generation.

112. Context compression

Context compression means reducing retrieved content while preserving useful information.

Example retrieved chunk:

This policy document applies to all employees across India offices. 
Annual leave is granted based on employment type. 
Employees may carry forward up to 5 unused annual leave days. 
Carry-forward requests must be submitted before March 31. 
The HR team reserves the right to reject incomplete requests.

Compressed context:

Employees may carry forward up to 5 unused annual leave days. 
Requests must be submitted before March 31.

Why compress?

Fit more useful evidence
Reduce cost
Reduce noise
Improve answer focus

But compression must be careful.

Bad compression may remove important conditions.

Example important condition:

Only confirmed full-time employees are eligible.

If removed, the answer may become wrong.

113. Lost-in-the-middle problem

The lost-in-the-middle problem means LLMs may pay more attention to information at the beginning and end of a long context, and may miss important information in the middle.

Example context:

Chunk 1: General leave introduction
Chunk 2: Eligibility
Chunk 3: Carry-forward rule  ← important
Chunk 4: Encashment
Chunk 5: Approval workflow
Chunk 6: Holiday calendar

If the important answer is buried in the middle, the LLM may miss it.

To reduce this problem:

Put most important chunks early
Avoid too much irrelevant context
Use clear source labels
Use shorter focused context
Repeat critical answer rules carefully

This is another reason context building matters.

114. Context ordering

Context ordering means deciding the sequence of chunks in the prompt.

Possible ordering strategies:

Most relevant first
Most recent first
Document order
Source authority first
Question subtopic order
Grouped by source
Grouped by answer structure

Example question:

Compare leave policy for interns and full-time employees.

Good context order:

1. Full-time employee leave policy
2. Intern leave policy
3. Any comparison/exception notes

Bad context order:

1. Travel policy
2. Intern leave policy
3. Payroll policy
4. Full-time employee leave policy

Good ordering makes generation easier and reduces mistakes.

116. System instruction

The system instruction defines the role and boundaries of the assistant.

Example:

You are an HR policy assistant.

Do not invent policy details.
If the answer is not found, say so clearly.

This tells the model:

What role it is playing
What source it should trust
What not to do
How cautious it should be

For enterprise RAG, the system instruction must be strict.

Example:

Do not answer from general knowledge.
Do not reveal information outside the user's access.
Do not cite a source unless it is present in the context.

117. User question

The user question is the actual thing the user wants answered.

Example:

Can I carry forward unused annual leave?

In a RAG prompt, the user question is usually placed after the retrieved context.

Why?

Because we want the model to read the evidence and then answer the question.

Example:

Context:
Employees may carry forward up to 5 unused annual leave days.

Question:
Can I carry forward unused annual leave?

The question should be clear.

If the question is unclear, query rewriting or clarification may be needed.

118. Retrieved context

Retrieved context is the evidence found by the RAG system.

Example:

[Source: HR Policy 2026, Section 4.3]
Employees may carry forward up to 5 unused annual leave days into the next calendar year.
Carry-forward requests must be submitted before March 31.

The retrieved context should ideally include:

Relevant chunk text
Source name
Section/page
Date/version
Metadata if useful

The LLM should answer from this context.

In RAG, the retrieved context is like an open-book reference.

Retrieved context = textbook page opened before answering

119. Answer rules

Answer rules tell the LLM how to respond.

Examples:

Answer clearly and briefly.
Use only the context.
Do not invent missing facts.
Mention any uncertainty.
Include citations.
If documents conflict, explain the conflict.
If answer is missing, say it is not found.

Answer rules are important because different use cases need different behavior.

For HR:

Be precise and policy-based.

For training assistant:

Explain step by step in simple language.

For legal RAG:

Do not give legal conclusion beyond the provided clauses.

For technical support:

Give steps, warnings, and source references.

120. Citation instruction

Citation instruction tells the model how to show sources.

Example:

After each key claim, cite the source document and section.

Or:

At the end of the answer, list the sources used.

Example answer:

Employees may carry forward up to 5 unused annual leave days. 
Carry-forward requests must be submitted before March 31.

Sources:
- HR Policy 2026, Section 4.3
- HR Policy 2026, Section 4.4

Good citation instruction prevents vague source display.

Bad citation:

Source: policy document

Better citation:

Source: HR Policy 2026, Section 4.3, Page 12

121. “Use only the provided context”

This is one of the most important RAG instructions.

Example:

Use only the provided context.
Do not answer from general knowledge.
If the context does not contain the answer, say:
"I could not find this in the provided documents."

Why this matters:

The LLM may know general information, but in RAG we want the answer to be based on the retrieved documents.

Example:

Question:

What is our company reimbursement limit?

General answer is dangerous.

The model should not say:

Usually companies reimburse up to ₹5,000.

It should answer only if the retrieved policy says the amount.

122. Handling missing answers

Sometimes retrieval does not find the answer.

A good RAG assistant should honestly say that.

Example:

I could not find the reimbursement limit in the provided documents.

A better answer:

I could not find the reimbursement limit in the provided context. 
The retrieved documents mention travel eligibility, but they do not specify a reimbursement amount.

This is better because it tells the user what was found and what was missing.

Bad answer:

The reimbursement limit is probably ₹10,000.

That is hallucination.

The fallback rule is essential.

123. Avoiding hallucination

Prompting helps reduce hallucination.

Useful rules:

Use only the provided context.
Do not guess.
Do not add facts not present in the context.
If the context is incomplete, say so.
If sources conflict, mention the conflict.
Quote or cite source sections where needed.

But prompt alone is not enough.

Hallucination can also come from:

Wrong retrieval
Missing documents
Bad chunking
Old documents
Conflicting context
Weak citation rules

So hallucination control needs both:

Good retrieval + good prompting

124. Asking clarifying questions

Sometimes the user question is ambiguous.

Example:

What is the policy?

Which policy?

Leave policy?
Travel policy?
Security policy?
Expense policy?
Work-from-home policy?

A good RAG assistant should ask a clarification instead of retrieving random documents.

Example response:

Which policy do you mean — leave, travel, expense, or work-from-home?

Another example:

Can I claim it?

The system needs to know what “it” refers to.

In chat, the assistant may use previous conversation context.

If previous question was about travel reimbursement, then:

Can I claim it?

may mean:

Can I claim travel reimbursement?

This connects to query rewriting.

125. Structured output in RAG

Structured output means asking the model to answer in a fixed format.

Example format:

Answer:
Reason:
Source:
Confidence:
Missing information:

Example:

The LLM receives a prepared prompt like this:

128. Grounded answer generation

Grounded answer generation means the answer is based on retrieved evidence.

Ungrounded answer:

Usually companies allow employees to carry forward 10 leaves.

Grounded answer:

Employees may carry forward up to 5 unused annual leave days, according to HR Policy 2026, Section 4.3.

The answer is grounded because it is connected to a source.

A grounded answer should not add unsupported information.

Example context:

Employees may carry forward up to 5 unused annual leave days.

Good answer:

Employees can carry forward up to 5 unused annual leave days.

Bad answer:

Employees can carry forward up to 5 unused annual leave days, but only after manager approval.

Why bad?

Because “manager approval” was not present in the retrieved context.

Unless the context says it, the LLM should not add it.

Some RAG systems include confidence.

High

Number of supporting chunks
Freshness of document
Whether context directly answers the question

Example:

Confidence is useful, but it should be explainable.

131. Answer with “not found” response

A good RAG system must know when not to answer.

Example question:

Good answer:

Bad answer:

The laptop replacement budget is probably ₹75,000.

The “not found” answer is very important.

It protects against hallucination.

A professional RAG assistant should say:

I found related information, but not the exact answer.

This is better than guessing.

137. Why RAG evaluation is difficult

RAG evaluation is difficult because there are two systems working together:

Retriever + LLM

So when the answer is wrong, we must ask:

Did retrieval fail?
Did generation fail?
Did chunking fail?
Did the prompt fail?
Was the source document wrong or outdated?

Example:

User asks:

What is the leave carry-forward limit?

Bad answer:

Employees can carry forward 10 days.

Possible reasons:

Old policy was retrieved.
Correct policy was not retrieved.
Both old and new policies were retrieved, but LLM picked the wrong one.
The prompt did not say to prefer the latest policy.
Metadata did not include document version.

So RAG evaluation is not just checking the final answer. We must evaluate the whole pipeline.

138. Retrieval quality

Retrieval quality means:

Did the system retrieve the right chunks?

Example question:

Can employees carry forward unused annual leave?

Good retrieval:

Chunk 1: Leave carry-forward rule
Chunk 2: Annual leave policy
Chunk 3: Carry-forward approval deadline

Bad retrieval:

Chunk 1: Holiday calendar
Chunk 2: Payroll process
Chunk 3: Travel policy

Retrieval quality depends on:

Chunking
Embeddings
Vector database
Top-k value
Metadata filtering
Query rewriting
Hybrid search
Re-ranking

Important:

If retrieval is bad, even a very strong LLM may answer badly.

139. Generation quality

Generation quality means:

Did the LLM produce a good answer from the retrieved context?

The retrieved context may be correct, but the LLM can still fail.

Example context:

Employees may carry forward up to 5 unused annual leave days.

Good generation:

Employees may carry forward up to 5 unused annual leave days.

Bad generation:

Employees may carry forward up to 10 unused annual leave days.

Here retrieval was correct, but generation failed.

Generation quality depends on:

Prompt template
Answer rules
LLM capability
Context order
Context length
Citation instruction
Temperature/settings

140. Faithfulness

Faithfulness means:

Is the answer faithful to the retrieved context?

A faithful answer does not add unsupported claims.

Context:

Employees may carry forward up to 5 unused annual leave days.

Faithful answer:

Employees may carry forward up to 5 unused annual leave days.

Unfaithful answer:

Employees may carry forward up to 5 unused annual leave days after manager approval.

Why unfaithful?

Because “after manager approval” was not in the context.

Faithfulness is one of the most important RAG evaluation measures.

Faithfulness checks whether the answer is grounded in the provided evidence.

141. Relevance

Relevance means:

Does the answer actually address the user’s question?

Question:

What is the carry-forward limit for annual leave?

Relevant answer:

The carry-forward limit is 5 unused annual leave days.

Irrelevant answer:

Annual leave is part of the company’s employee benefits policy.

The second answer may be true, but it does not answer the question.

Relevance applies to both:

Retrieved context relevance
Final answer relevance

142. Context precision

Context precision asks:

Out of the retrieved chunks, how many were actually useful?

Example:

Retriever returns 5 chunks:

Chunk 1: Leave carry-forward rule        useful
Chunk 2: Annual leave eligibility        useful
Chunk 3: Payroll date                    not useful
Chunk 4: Office parking                  not useful
Chunk 5: Holiday calendar                not useful

Useful chunks = 2 Total chunks = 5

Context precision = 2/5

High context precision means most retrieved chunks are useful.

Low context precision means the LLM is receiving noisy context.

Low precision = too much junk in context

143. Context recall

Context recall asks:

Did we retrieve all the information needed to answer correctly?

Example question:

Can I carry forward annual leave, and what is the deadline?

Needed chunks:

Chunk A: Carry-forward limit is 5 days.
Chunk B: Deadline is March 31.

If retriever gets only Chunk A, answer may be incomplete.

Context recall is low because deadline information was missed.

High context recall means the system found all necessary evidence.

Simple difference:

Context precision = Did we avoid useless chunks?
Context recall = Did we include all needed chunks?

Both matter.

144. Answer correctness

Answer correctness means:

Is the final answer factually correct?

Question:

How many leave days can be carried forward?

Correct answer:

5 days.

Wrong answer:

10 days.

But answer correctness is tricky because a final answer can be wrong for different reasons.

Wrong retrieval
Wrong source
Wrong generation
Outdated document
Missing context
Misread context

So when evaluating answer correctness, we should also inspect the retrieved chunks.

145. Hallucination rate

Hallucination rate measures how often the system gives unsupported or invented answers.

Example hallucinated answer:

The laptop replacement budget is ₹75,000.

But the retrieved context did not mention any laptop replacement budget.

A good RAG system should reduce hallucination by:

Using only retrieved context
Giving not-found responses
Showing citations
Keeping context focused
Avoiding unsupported claims

Hallucination rate is important in:

HR
Legal
Finance
Healthcare
Compliance
Enterprise policy systems

Because wrong confident answers can create real risk.

146. Source accuracy

Source accuracy means:

Are the citations correct?

Example answer:

Employees may carry forward up to 5 leave days.

Source: HR Policy 2026, Section 4.3

This is source-accurate only if Section 4.3 actually says that.

Bad source accuracy:

Answer is correct, but citation points to wrong section.

Example:

Answer: Carry-forward limit is 5 days.
Source: Holiday Calendar, Page 2

The answer may be right, but the source is wrong.

In enterprise RAG, wrong citation is a serious problem because users may verify the wrong document.

Wrong retrieved chunk:

Employees can claim hotel reimbursement up to ₹5,000 per night.

Wrong retrieval → wrong context → wrong answer

Poor keyword matching
Weak query rewriting
No metadata filter
Poor chunking
Too low top-k

153. Right document not retrieved

Sometimes the correct answer exists in the knowledge base, but the retriever fails to find it.

Example:

Employees may carry forward up to 5 unused annual leave days.

Can I move my unused vacation to next year?

The system may fail if it does not understand that:

vacation = annual leave
move to next year = carry forward

Solutions:

Better embedding model
Hybrid search
Query expansion
Query rewriting
Domain glossary
Better chunk metadata
Higher top-k

This is a very common RAG failure.

154. Chunk too small

If chunks are too small, they may lose meaning.

Bad small chunk:

up to 5 days

This is not enough.

The LLM does not know:

5 days of what?
Who is eligible?
Which policy?
What condition?

Better chunk:

Leave Carry Forward:
Employees may carry forward up to 5 unused annual leave days into the next calendar year.

Small chunks can improve precision, but if they are too small, they become meaningless.

155. Chunk too large

If chunks are too large, they contain too much unrelated information.

Example large chunk:

Leave policy + travel policy + payroll rules + parking rules + holiday list

User asks:

Can I carry forward annual leave?

The answer is somewhere inside the big chunk, but the LLM may be distracted.

Problems with large chunks:

More noise
Higher cost
Slower response
Weak citation precision
Lost-in-the-middle problem
Lower retrieval focus

Good chunking balances context and precision.

156. Duplicate chunks

Duplicate chunks happen when the same or similar text appears many times.

Example:

Chunk 1: Employees may carry forward up to 5 unused annual leave days.
Chunk 2: Employees may carry forward up to 5 unused annual leave days.
Chunk 3: Annual leave carry-forward is limited to 5 days.

Duplicates waste context space.

They can also make the model overemphasize one point.

Common causes:

Repeated headers/footers
Multiple document versions
Copied policy sections
Email thread repetition
Overlapping chunks
Poor deduplication

Solution:

Deduplicate chunks before final context building.

157. Old data retrieved

This is a serious enterprise RAG problem.

Example:

Old policy:

Employees may carry forward 10 leave days.

New policy:

Employees may carry forward 5 leave days.

If the system retrieves the old policy, the answer becomes wrong.

Solutions:

Store document version
Store effective date
Prefer latest documents
Archive old documents
Use freshness filters
Show source date in citation

Important rule:

RAG must know not only what a document says,
but also whether that document is current.

158. Bad OCR

It is used when documents are scanned images instead of selectable text.

Bad OCR can corrupt the text.

Original scanned document:

Now retrieval may fail because the text is damaged.

Bad OCR affects:

Search quality
Embedding quality
Citation quality
Answer correctness

Solutions:

Use better OCR
Manually review important documents
Clean extracted text
Use layout-aware extraction
Avoid indexing unreadable scans blindly

159. Poor metadata

Section: Leave Carry Forward
Page: 12
Version: v3
Department: HR
Region: India
Security Level: Internal

Poor metadata:

Document: file1.pdf
Page: unknown
Version: unknown
Department: unknown

Poor metadata causes problems:

Cannot filter by department
Cannot prefer latest version
Cannot show good citations
Cannot enforce access control
Cannot debug retrieval easily

In enterprise RAG, metadata is not optional. It is part of the quality system.

160. Ambiguous user query

Users often ask unclear questions.

Example:

This is ambiguous.

Possible meanings:

Leave policy?
Travel policy?
Expense policy?
Security policy?
Work-from-home policy?

Another example:

What is “it”?

A good RAG system should not guess blindly.

Better response:

Which policy do you mean — leave, travel, expense, security, or work-from-home?

Or, if conversation context is available, the system can rewrite the query.

161. LLM ignores context

Sometimes the correct context is retrieved, but the LLM still answers from general knowledge.

Context:

Employees may carry forward up to 5 unused annual leave days.

Bad answer:

Most companies allow 10 to 15 days to be carried forward.

This means the LLM ignored the context.

Solutions:

Stronger system instruction
Use-only-context rule
Citation requirement
Lower temperature
Better prompt structure
Put important context clearly
Add not-found rule

This is a generation failure, not retrieval failure.

162. LLM hallucinates

Hallucination means the model invents unsupported information.

Context:

Employees may carry forward up to 5 unused annual leave days.

Hallucinated answer:

Employees may carry forward up to 5 days, but only after HR manager approval.

The “HR manager approval” part was invented.

Why hallucination happens:

Prompt is weak
Context is incomplete
User asks for missing information
LLM tries to be helpful
Sources conflict
Model uses general knowledge

Good RAG design should force the model to say:

I could not find that detail in the provided documents.

163. Source mismatch

Source mismatch means the answer and citation do not match.

Example answer:

Employees may carry forward up to 5 leave days.

Wrong source:

Source: Holiday Calendar 2026, Page 2

The answer may be correct, but the citation is wrong.

This is dangerous because users may verify the cited source and not find the answer.

Source mismatch can happen when:

Source labels are missing
Chunks are combined badly
LLM invents citations
Context builder loses metadata
Multiple chunks are mixed

Solution:

Attach source metadata tightly to each chunk.
Do not allow the LLM to invent source names.

164. Conflicting documents

Sometimes retrieved documents disagree.

Example:

HR Policy 2023:
Employees may carry forward 10 leave days.

HR Policy 2026:
Employees may carry forward 5 leave days.

A bad answer silently chooses one.

A better answer says:

The retrieved documents conflict. The 2023 policy says 10 days, while the 2026 policy says 5 days. Since the 2026 policy is newer, it is likely the current rule, but this should be confirmed with HR.

Conflicts happen because of:

Old and new versions
Regional policy differences
Draft vs approved document
Different departments
Unclear effective dates

Good RAG should detect and explain conflicts.

165. Security leakage

Security leakage is one of the most serious RAG failures.

It means the system retrieves or reveals information the user is not allowed to see.

Example:

A contractor asks:

What is the executive salary policy?

If the system retrieves confidential HR data and answers, that is a security failure.

Security leakage can happen because:

No access control
Metadata filters missing
Vector DB stores all chunks together
Application checks permission after retrieval instead of before retrieval
Citations expose restricted file names

Correct approach:

Apply permissions before retrieval.
Retrieve only documents the user is allowed to access.
Log access.
Protect sensitive metadata.

Important rule:

Enterprise RAG must be permission-aware.

RAG Failure Diagnosis

When a RAG answer is wrong, ask:

1. Was the right document available?
2. Was the document extracted correctly?
3. Was the chunking good?
4. Was metadata complete?
5. Did retrieval find the right chunks?
6. Did context builder remove noise?
7. Did the prompt force source-based answering?
8. Did the LLM follow the context?
9. Were citations correct?
10. Was access control respected?

Step 11 Summary

RAG can fail at many points.

Bad documents create bad chunks.
Bad chunks create bad embeddings.
Bad embeddings create bad retrieval.
Bad retrieval creates bad context.
Bad context creates bad answers.
Weak prompting allows hallucination.
Poor metadata causes citation, freshness, and security problems.

The most important lesson:

RAG quality is pipeline quality.

It is not enough to say:

Use vector DB + LLM.

A serious RAG system needs good document preparation, retrieval, context building, prompting, evaluation, and security.

User question
   ↓
Embed question
   ↓
Vector search
   ↓
Take top-k chunks
   ↓
Put chunks in prompt
   ↓
LLM answers

This is good for learning and small demos.

Example:

Question:
What is chunking?

Retriever:
Finds 3 chunks about chunking.

LLM:
Explains chunking using those chunks.

But naive RAG has limits.

It can fail when:

Question is complex

Multiple sources are needed
Metadata matters
Documents conflict
User needs latest data
Query needs rewriting
Retrieved chunks are noisy

Naive RAG is a starting point, not the final production architecture.

168. Modular RAG

Modular RAG means the RAG pipeline is divided into replaceable modules.

Example modules:

Document loader
Text extractor
Chunker
Embedding model
Vector store
Retriever
Re-ranker
Context builder
Prompt builder
LLM
Citation generator
Evaluator
Logger

Why modular?

Because we can improve one part without rewriting everything.

Example:

Current embedding model is weak.
Replace only embedding model.

Current retriever is noisy.
Add re-ranker.

Current chunking is bad.
Replace chunking module.

Modular RAG is good for production because RAG systems evolve.

169. Agentic RAG

Agentic RAG means an agent controls the retrieval process instead of using one fixed retrieval step.

In normal RAG:

Retrieve once → answer

In Agentic RAG:

Think about question
Choose search strategy
Retrieve information
Check if enough
Maybe retrieve again
Maybe use another tool
Then answer

Example:

User asks:

Compare our 2025 and 2026 leave policies and explain what changed.

Agentic RAG may do:

1. Search for 2025 leave policy
2. Search for 2026 leave policy
3. Compare relevant sections
4. Detect differences
5. Generate answer with sources

Agentic RAG is useful for complex questions, but it must be controlled because agents can increase cost and latency.

170. Corrective RAG

Corrective RAG means the system checks whether retrieved context is good enough.

If retrieval is poor, it corrects the process.

Flow:

Retrieve chunks
   ↓
Check relevance
   ↓
If chunks are good → answer
   ↓
If chunks are bad → rewrite query / retrieve again / use web or another source

Example:

User asks:

What is the refund rule for premium customers?

First retrieval returns general product description, not refund policy.

Corrective RAG detects:

Retrieved chunks do not answer the question.

Then it retries with:

premium customer refund policy
refund terms premium plan

Corrective RAG reduces bad answers caused by weak retrieval.

Adaptive RAG changes its strategy depending on the question.

Not every question needs the same pipeline.

Example:

Question TypeBest Strategy
Simple factTop-k retrieval
Exact ID/codeKeyword or hybrid search
ComparisonMulti-query retrieval
Current dataFreshness filter
SQL countSQL tool
Complex reasoningAgentic or multi-hop retrieval
Ambiguous queryClarifying question

Example:

Question:

Use keyword/hybrid search because PX-104 is exact.

Question:

What should I do if login keeps timing out?

Use vector/hybrid search because wording may vary.

Adaptive RAG means:

Choose the retrieval method based on the question.

3. Compare

Multi-hop RAG is stronger than one-shot retrieval for complex questions.

176. Re-ranking

Re-ranking means sorting retrieved chunks again after initial retrieval.

Initial retriever may return 20 chunks.

Then a re-ranker selects the best 5.

Flow:

Initial retrieval → candidate chunks → re-ranker → best chunks → LLM

Why useful?

Vector search may retrieve chunks that are semantically close but not truly answering the question.

Example question:

What is the carry-forward deadline?

Initial retrieval:

Chunk 1: carry-forward limit
Chunk 2: annual leave eligibility
Chunk 3: carry-forward deadline
Chunk 4: leave approval workflow

Re-ranker can move Chunk 3 to the top because it directly answers the deadline question.

Re-ranking improves context quality.

177. Cross-encoder re-ranker

A cross-encoder re-ranker compares the question and chunk together.

A normal vector search compares precomputed vectors.

A cross-encoder reads both together:

Question + Chunk → relevance score

Example:

Question:

What is the deadline for carry-forward request?

Chunk A:

Employees may carry forward 5 days.

Chunk B:

Carry-forward requests must be submitted before March 31.

Cross-encoder can score Chunk B higher because it directly answers the question.

Trade-off:

More accurate
But slower and more expensive than vector search

Common production pattern:

Retrieve top 20 with vector/hybrid search
Re-rank to top 5 with cross-encoder
Send top 5 to LLM

178. Query decomposition

Query decomposition means breaking a complex question into smaller questions.

Example:

Compare leave policy for interns and full-time employees and mention approval process.

Break into:

1. What is the leave policy for interns?
2. What is the leave policy for full-time employees?
3. What is the leave approval process?
4. Compare the differences.

Why useful?

Because one retrieval query may not find all needed information.

Query decomposition improves multi-part answers.

It is useful for:

Comparisons
Summaries
Root cause analysis
Policy analysis
Multi-document answers
Research-style questions

182. Access control

Example:

Identify user

Generate answer from allowed chunks

This is one of the most important comparisons.

Many people think:

If the model does not know my data, fine-tune it.

Fine-tuning means

Fine-tuning means training the model further so that its behavior changes.

It is useful when you want the model to learn:

A style
A format
A classification pattern
A domain-specific response pattern
A repeated task behavior

Example:

Always classify support tickets into:
Billing, Technical, Access, Cancellation

Fine-tuning can help here.

RAG means

RAG does not change the model.

RAG gives the model relevant reference material at answer time.

It is useful when answers must come from:

Latest documents
Private documents
Policies
Manuals
Contracts
Knowledge base articles
Reports

Example:

What is our latest leave policy?

This should use RAG, not fine-tuning.

Why?

Because policies change. It is better to update the document index than retrain the model repeatedly.

Simple difference

Fine-tuning = change model behavior
RAG = give model external knowledge
Need Better Choice
Answer from latest HR policy RAG
Answer from private company PDFs RAG
Show source citations RAG
Update answer when document changes RAG
Follow a fixed output style Fine-tuning / prompting
Classify fixed categories Fine-tuning / prompting
Learn a company tone Fine-tuning / prompting

197. RAG vs prompt engineering

Prompt engineering means writing better instructions for the model.

Example prompt:

Answer in simple language.
Use bullet points.
Do not guess.
Ask clarification if needed.

Prompt engineering controls how the model responds.

But prompt engineering alone does not give the model private or latest knowledge.

Example:

What is our company laptop replacement policy?

A good prompt cannot magically know your company policy.

RAG adds the missing knowledge.

So:

Prompt engineering = better instructions
RAG = better information + instructions

In real RAG systems, we use both.

RAG retrieves the context.
Prompt engineering tells the LLM how to use the context.

Example RAG prompt:

Use only the provided context.
If the answer is not in the context, say it is not found.
Cite the source section.

This is prompt engineering inside RAG.

198. RAG vs function calling

Function calling means the LLM calls a tool or API.

Example tools:

Get weather
Query database
Book calendar event
Send email
Calculate amount
Fetch order status
Create ticket

RAG retrieves knowledge from documents.

Function calling performs an action or fetches structured/live data.

Example RAG question:

What does the refund policy say?

RAG retrieves refund policy chunks and answers.

Example function calling question:

What is the refund status of order #12345?

The model may call an order API.

RAG + function calling together

Many real systems use both.

Example:

Question:
Based on the refund policy, how much refund should order #12345 receive?

System may do:

RAG:
Retrieve refund policy.

Function call:
Fetch order amount and cancellation date.

Calculator:
Compute refund amount.

LLM:
Generate final answer with source.

So:

RAG = retrieve knowledge
Function calling = use tools/actions/live systems

199. RAG vs agents

An agent is a system where the LLM can plan steps, choose tools, and work toward a goal.

RAG is usually one pattern inside an agent.

Simple RAG:

Question → retrieve → answer

Agent:

Goal → plan → choose tools → retrieve/search/calculate/act → check result → answer

Example RAG question:

What is the leave carry-forward rule?

Simple RAG is enough.

Example agentic question:

Compare 2025 and 2026 leave policies, identify what changed, draft an email to employees, and create a task for HR review.

This may need an agent because it involves multiple steps:

Retrieve 2025 policy
Retrieve 2026 policy
Compare
Draft email
Maybe create task
Maybe ask approval

So:

RAG = knowledge retrieval + answer
Agent = multi-step planning + tool use + possibly RAG

200. RAG vs semantic search

Semantic search means searching by meaning.

Example:

User searches:

vacation carry forward

The system finds:

annual leave rollover policy

Even though words differ, meaning is similar.

Semantic search usually returns documents or chunks.

RAG goes one step further.

Semantic search = find relevant content
RAG = find relevant content + generate answer from it

Example semantic search result:

Top matching sections:
1. HR Policy 2026, Leave Carry Forward
2. Annual Leave Rules
3. Leave Approval Workflow

Example RAG answer:

Employees may carry forward up to 5 unused annual leave days. Requests must be submitted before March 31.

Sources:
- HR Policy 2026, Section 4.3
- HR Policy 2026, Section 4.4

So RAG often uses semantic search internally.

201. RAG vs traditional search

Traditional search usually depends heavily on keywords.

Example search:

notice period resignation

It finds documents containing those terms.

Traditional search is good for:

Exact keywords
Document lookup
IDs
Names
Titles
Known phrases

But it can fail when the wording changes.

Example user says:

How long do I need to work after quitting?

Document says:

Employees must serve a 60-day notice period after resignation.

Traditional keyword search may not match well.

RAG with embeddings can understand the meaning better.

Simple difference:

Traditional search = keyword-based document finding
Semantic search = meaning-based document finding
RAG = search + answer generation

202. RAG vs knowledge graph

A knowledge graph stores entities and relationships.

Example:

Employee → belongs to → Department
Policy → applies to → Region
Product → has → Warranty
Customer → has → Plan
Plan → has → SLA

Knowledge graphs are excellent when relationships matter.

Example question:

Which security policies apply to contractors working on finance systems?

A knowledge graph can trace relationships:

Contractor
   ↓
works on finance system
   ↓
requires security access
   ↓
applicable policies

RAG is excellent when answers must come from documents.

Example:

Explain the security policy section that applies to contractors.

Best approach can be:

Knowledge graph finds the related entities and policies.
RAG retrieves the policy text.
LLM explains it with sources.

So:

Knowledge graph = structured relationships
RAG = source-based natural language answer

They can work together.

204. When not to use RAG

Do not use RAG for everything.

RAG may not be needed when:

The answer is simple and already in the prompt
The task is pure writing or rewriting
The task is creative generation
The task needs calculation, not documents
The task needs live API data
The task needs a database transaction
The data is small enough to fit directly in prompt
There is no reliable document source

First fix the knowledge base.

Do not use RAG just because it sounds advanced.
